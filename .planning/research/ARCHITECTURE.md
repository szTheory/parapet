### Component Responsibilities

| Component | Responsibility | Implementation |
|-----------|----------------|----------------|
| `Parapet` (host-generated) | Top-level config module; declares surfaces, SLOs, exporters | `use Parapet, otp_app: :my_app` macro; generated by `mix parapet.install` |
| `Parapet.Supervisor` | OTP supervision tree; owns telemetry handler lifecycle | GenServer-based supervisor started in host app's `Application` |
| `Parapet.Telemetry` | Attaches `:telemetry` handlers; normalizes and routes events | Handler attach/detach lifecycle; must never block or do I/O |
| `Parapet.Metrics` | Generates `Telemetry.Metrics` specs with enforced label contracts | Returns metric definitions consumed by reporter adapters |
| `Parapet.Phoenix` | Phoenix endpoint/router integration; route classification | Listens to `[:phoenix, :router_dispatch, :stop]` events |
| `Parapet.Oban` | Job metrics, queue SLOs, retry-aware semantics | Listens to `[:oban, :job, :stop]` and `[:oban, :job, :exception]` |
| `Parapet.Ecto` | DB query/pool metrics; distinguishes queue vs query time | Listens to `[:my_app, :repo, :query]` events |
| `Parapet.SLO` | SLI/SLO model; burn-rate math; Prometheus rule generation | Pure functions; generates recording + alert rules |
| `Parapet.Dashboards` | Produces Grafana JSON and Prometheus rule files | Outputs to `priv/parapet/grafana/` and `priv/parapet/prometheus/` |
| `Parapet.Doctor` | Lints config, labels, security posture; CI-gate mode | `mix parapet.doctor [--ci]` |
| `Parapet.Test` | ExUnit helpers; telemetry assertions; SLO fixture helpers | `import Parapet.Test` in test support |
| `Parapet.Sigra` (optional) | Auth success/failure SLO integration; compiles out if absent | Conditional compile via `optional_dep: :sigra` |

## Recommended Project Structure

```
lib/
└── parapet/
    ├── application.ex          # (library) starts Parapet.Supervisor
    ├── supervisor.ex           # OTP supervisor; owns handler lifecycle
    ├── telemetry.ex            # handler attach/detach; event normalization
    ├── metrics.ex              # Telemetry.Metrics spec generation
    ├── slo/
    │   ├── slo.ex              # SLO struct; burn-rate math
    │   ├── rule_generator.ex   # Prometheus recording + alert rules
    │   └── openslo.ex          # OpenSLO YAML export
    ├── surfaces/
    │   ├── phoenix.ex          # route classification; HTTP SLI
    │   ├── oban.ex             # job/queue SLI; retry-aware
    │   └── ecto.ex             # DB pool/query SLI
    ├── integrations/
    │   └── sigra.ex            # optional; compiles out without :sigra dep
    ├── dashboards/
    │   ├── grafana.ex          # Grafana JSON generation
    │   └── prometheus.ex       # Prometheus rule file output
    ├── doctor/
    │   ├── doctor.ex           # mix parapet.doctor orchestrator
    │   ├── checks/
    │   │   ├── cardinality.ex  # label cardinality linting
    │   │   ├── security.ex     # endpoint/dashboard exposure checks
    │   │   └── config.ex       # SLO completeness, deploy ID, etc.
    ├── label_policy.ex         # canonical low-cardinality label enforcement
    ├── redaction.ex            # redact-at-emission for telemetry metadata
    └── test.ex                 # ExUnit helpers

priv/
└── parapet/
    ├── grafana/
    │   └── dashboards/         # generated *.json (host-owned after generation)
    └── prometheus/
        └── rules/              # generated *.yml (host-owned after generation)

mix.exs                         # single :parapet package; files: whitelist
```

### Structure Rationale

- **`lib/parapet/surfaces/`:** Groups telemetry integrations by ecosystem surface (Phoenix, Oban, Ecto), not by concern (metrics vs SLO). Each surface owns all knowledge about its event names, label contracts, and SLI calculations — keeping those facts in one place prevents drift.
- **`lib/parapet/integrations/`:** Optional sibling-library integrations live here, separate from surfaces. Each file must compile to a no-op when the optional dep is absent. Never leak into surfaces.
- **`lib/parapet/slo/`:** SLO structs, math, and output generation are isolated from telemetry concerns. This separation allows SLOs to be defined and validated at compile time, before any runtime events arrive.
- **`priv/parapet/`:** All generated artifacts go here so the host app's `git status` reveals what Parapet generated. Host owns these files after generation — Parapet never overwrites without confirmation.
- **`lib/parapet/doctor/`:** Doctor checks are first-class modules, not afterthoughts. Separating check types (cardinality, security, config) lets the CI gate be precise about severity.

## Architectural Patterns

### Pattern 1: Telemetry Handler — Fast, Non-Blocking, Non-I/O

**What:** Attach `:telemetry` handlers that do exactly three things: classify the event, normalize it into Parapet's internal shape, and enqueue/emit to reporters. Return immediately.

**When to use:** Every telemetry integration. This is the hot path — runs on every Phoenix request, every Oban job, every Ecto query.

**Trade-offs:** Handlers that crash detach silently — the BEAM drops the handler rather than crashing the caller. This is a real footgun: losing handlers means losing metrics without any visible error. Parapet must emit `parapet_handler_attached_total` and `parapet_handler_detached_total` metrics so handler health is observable.

```elixir
def handle_event([:phoenix, :router_dispatch, :stop], measurements, metadata, _config) do
  labels = classify_route(metadata)
  emit_http_metrics(measurements, labels)
  :ok
end

# NEVER do this inside a handler:
# HTTPoison.post(...)     # network I/O
# Repo.insert(...)        # DB call
# Logger.error(...)       # even this can block under load
```

---

### Pattern 2: Compile-Time SLO Validation + Runtime Event Counting

**What:** SLO definitions are validated at compile time (event names exist, objectives are sane, labels are low-cardinality). At runtime, telemetry events increment two counters per SLO: `parapet_slo_good_total` and `parapet_slo_total`. Prometheus computes the ratio and burn rate from those counters.

**When to use:** Every SLO definition. The key invariant: the library computes the counter contract at compile time, so runtime events are cheap increments, not expensive ratio computations.

**Trade-offs:** Compile-time validation catches configuration errors before deployment, not at alert-fire time. The tradeoff is that SLO changes require a recompile, which is correct — SLO changes are code changes and deserve a CI run.

```elixir
defmodule MyApp.Parapet do
  use Parapet, otp_app: :my_app

  slo :api_availability do
    surface :http
    objective 99.9
    window :days_30
    good_event [:phoenix, :router_dispatch, :stop], where: [status_class: "2xx"]
    total_event [:phoenix, :router_dispatch, :stop]
    alert :fast_burn, notify: :page
    alert :slow_burn, notify: :ticket
  end
end
```

---

### Pattern 3: Generator for Scaffolding, Library for Runtime

**What:** `mix parapet.install` generates host-owned config and artifact files. The generated `lib/my_app/parapet.ex` is the single point of truth for what Parapet does in this host app. The library's runtime behavior reads from this module, not from `config/*.exs`.

**When to use:** Every Parapet installation. This pattern is non-negotiable — it's what makes Parapet inspectable, forkable, and adoptable by teams skeptical of library magic.

**Trade-offs:** Generated files require the host to commit them, review them, and update them when Parapet is upgraded. This is a feature, not a bug — it forces a deliberate upgrade path and prevents silent drift.

```elixir
# Generated by mix parapet.install — host-owned, modify freely
defmodule MyApp.Parapet do
  use Parapet,
    otp_app: :my_app,
    endpoint: MyAppWeb.Endpoint,
    router: MyAppWeb.Router,
    repo: MyApp.Repo

  service_version from: {:env, "RELEASE_VERSION"}
  deployment_id from: {:env, "RELEASE_SHA"}

  exporters do
    prometheus path: "/metrics", protect: :private_network
  end
end
```

---

### Pattern 4: Optional Dependency Seams

**What:** Every integration dep (`:sigra`, `:oban`) is declared as an optional Mix dependency. The integration module uses `Code.ensure_loaded?/1` at the module level (or a compile-time conditional) to decide whether to define its handler functions. Parapet's supervisor conditionally starts these handlers.

**When to use:** Every sibling-library integration and every optional external dep. Failure to follow this pattern means base installs break for users who don't have the optional dep.

**Trade-offs:** Slightly more indirection in supervisor setup. Worth it: base install must work with zero optional deps, and `mix parapet.doctor` can detect when a dep is absent but the feature is configured.

```elixir
# lib/parapet/integrations/sigra.ex
if Code.ensure_loaded?(Sigra) do
  defmodule Parapet.Integrations.Sigra do
    def attach_handlers(config), do: ...
  end
else
  defmodule Parapet.Integrations.Sigra do
    def attach_handlers(_config), do: :noop
  end
end
```

---

### Pattern 5: Low-Cardinality Labels at Emission, High-Cardinality in Wide Events

**What:** Metric labels are validated against a hardcoded allowlist of low-cardinality values. Any label not on the allowlist (e.g., `user_id`, `request_id`, `account_id`) triggers a compile-time warning and a doctor check failure. High-cardinality values belong in structured wide events emitted to the logger, not in Prometheus labels.

**When to use:** Every metric definition. This is the most common footgun in Phoenix telemetry setups — raw `:path` labels in Prometheus cause unbounded time series and crash Prometheus memory.

**Trade-offs:** Developers sometimes want per-user or per-account metrics. The right answer is always: use Prometheus for aggregate health, use wide events + traces for per-user investigation.

```elixir
# Correct — low cardinality
[:route, :method, :status_class, :queue, :worker]

# Wrong — high cardinality, will be caught by Doctor
[:user_id, :account_id, :request_id, :email, :path]
```

## Data Flow

### Request Telemetry Flow

```
Phoenix Request
    ↓
[:phoenix, :router_dispatch, :stop] ← emitted by Phoenix
    ↓
Parapet.Telemetry handler (attached at app startup)
    ↓ classify route (uses compiled route table)
    ↓ normalize labels (apply low-cardinality policy)
    ↓ redact metadata (at emission, not at storage)
    ↓
Parapet.Metrics.emit/2 (increments Telemetry.Metrics counters)
    ↓
Reporter (telemetry_metrics_prometheus / PromEx / custom)
    ↓
Prometheus scrapes /metrics endpoint
    ↓
Grafana evaluates recording rules → dashboard panels
Alertmanager evaluates alert rules → pages / tickets
```

### SLO Burn Rate Flow

```
compile time:
  Parapet.SLO.validate!()
  → Parapet.Dashboards.Prometheus.generate_rules()
  → priv/parapet/prometheus/rules/slo_api_availability.yml

runtime:
  each request → parapet_slo_good_total{slo="api_availability"} ++
  each request → parapet_slo_total{slo="api_availability"} ++

Prometheus evaluates recording rules (5m, 1h windows):
  error_ratio = 1 - (good_rate / total_rate)
  burn_rate = error_ratio / (1 - objective)

Alert fires when burn_rate > 14.4 (fast burn, 5m + 1h windows agree)
```

### Key Data Flows

1. **Handler attach flow:** `Parapet.Supervisor` starts → calls `Parapet.Telemetry.attach_all/1` → handlers registered with `:telemetry.attach/4` using stable handler IDs based on the host module name (prevents double-attach on hot reload).

2. **Generator flow:** `mix parapet.install` → reads host app's `mix.exs` to detect Phoenix/Ecto/Oban → writes `lib/my_app/parapet.ex`, `priv/parapet/prometheus/rules/*.yml`, `priv/parapet/grafana/dashboards/*.json` → each file is host-owned after generation.

3. **Doctor flow:** `mix parapet.doctor` → loads host `Parapet` module → inspects SLO definitions, metric specs, runtime config, OTP application environment → emits structured check results → optionally exits non-zero for CI gate.

4. **Deploy marker flow:** `RELEASE_SHA` env var read at startup → stored in a module attribute or ETS → attached to every wide event and SLO window annotation — this is what enables "error spike started 3 minutes after deploy abc123."

## Scaling Considerations

| Scale | Architecture Concern |
|-------|---------------------|
| Single-node SaaS (most adopters) | Telemetry handler overhead is the only concern; keep handlers below 5µs per event |
| Multi-node / distributed | Each node attaches its own handlers; Prometheus scrapes each node; Grafana aggregates — no Parapet-specific distributed state needed |
| High-request-volume | Route classification must use compiled lookup, not dynamic regex per request; pre-compile the route table at startup |
| Many SLOs / many metrics | Each additional SLO adds 2 counters; Prometheus handles thousands of time series without issue; cardinality from *labels* is the real risk |

### Scaling Priorities

1. **First bottleneck:** Telemetry handler latency on high-throughput endpoints. Fix by moving any non-trivial work (route regex matching) into a compile-time lookup table, not a runtime computation.

2. **Second bottleneck:** Prometheus scrape duration growing due to high-cardinality labels. Fix by enforcing the label allowlist — this should never happen if doctor checks pass.

## Anti-Patterns

### Anti-Pattern 1: Blocking in Telemetry Handlers

**What people do:** Put DB inserts, HTTP calls, or heavy computation inside `:telemetry` handler callbacks to "capture evidence."

**Why it's wrong:** Telemetry handlers run synchronously on the calling process. Blocking a handler blocks the Phoenix request or Oban job that emitted the event. Slow handlers also detach silently when they crash under BEAM pressure, causing metric loss with no visible error.

**Do this instead:** Handlers normalize and enqueue. All I/O goes through async workers (GenServer / Task / Oban job). Evidence capture belongs in a v0.2 durable spine, not in v0.1 handlers.

---

### Anti-Pattern 2: High-Cardinality Metric Labels

**What people do:** Tag Prometheus metrics with `:path` (raw URL), `:user_id`, `:account_id`, or `:status` (numeric, 200+ distinct values).

**Why it's wrong:** Prometheus creates one time series per unique label combination. A Phoenix app with 50 routes × 1000 users × 5 status codes = 250,000 time series. This crashes Prometheus memory and makes cardinality-based billing extremely expensive.

**Do this instead:** Use `:route` (Phoenix route pattern, e.g., `/users/:id`), `:method`, `:status_class` (`2xx`/`4xx`/`5xx`). Route patterns are bounded by the number of routes in the router, which is compile-time constant.

---

### Anti-Pattern 3: Conflating Telemetry Events With Durable Evidence

**What people do:** Treat `:telemetry` events as the evidence store — subscribing handlers that write to DB on every event, trying to replay event history from telemetry.

**Why it's wrong:** Telemetry is ephemeral and lossy. If a handler crashes, events are lost. If the app restarts, in-flight events are gone. Telemetry is a real-time signal, not an audit log.

**Do this instead:** v0.1 uses telemetry-only. v0.2 introduces a durable evidence spine (Ecto-backed). Design the event contract now so it doesn't break when persistence is added — that is the core architectural insight of Parapet.

---

### Anti-Pattern 4: Double-Attaching Telemetry Handlers

**What people do:** Attach handlers in `Application.start/2` without checking if they're already attached, causing duplicate metrics on hot code reload in development.

**Why it's wrong:** Each handler attachment is additive — duplicate handlers cause double-counting, which silently corrupts SLO math.

**Do this instead:** Use stable, deterministic handler IDs based on the host module name (e.g., `{MyApp.Parapet, :phoenix_http}`). Parapet.Telemetry calls `:telemetry.detach/1` before `:telemetry.attach/4` to make attachment idempotent.

---

### Anti-Pattern 5: Metrics Endpoint Public in Production

**What people do:** Mount `telemetry_metrics_prometheus` on the default port (9568) or as a Phoenix route without auth, then ship to production.

**Why it's wrong:** `/metrics` exposes application internals — route names, job queue depths, DB pool states, error rates — to unauthenticated external clients. This is an information disclosure vulnerability.

**Do this instead:** `mix parapet.doctor` flags this as a hard error in production. Document three safe patterns: private network port, Phoenix route + auth plug, sidecar exporter. Default generated config leaves endpoint protection explicit and uncommitted until the host configures it.

---

### Anti-Pattern 6: Premature Package Split

**What people do:** Split into `parapet` + `parapet_admin` + `parapet_oban` before the API is stable, forcing adopters to manage multiple packages.

**Why it's wrong:** Package splits create coordination cost and version skew. Adopters must align versions across packages. Before the public API is stable, this creates unnecessary friction.

**Do this instead:** Single `parapet` package for v0.1. Design the internal module boundary between `Parapet.*` (core) and future `ParapetAdmin.*` (UI) at the code level now, but keep them in one package until the split pays for itself.

## Integration Points

### External Systems

| System | Integration Pattern | Notes |
|--------|---------------------|-------|
| Prometheus | Generated recording/alert rules in `priv/parapet/prometheus/rules/`; Parapet attaches a reporter or defers to PromEx | Host configures scrape target; Parapet never owns the Prometheus server |
| Grafana | Generated dashboard JSON in `priv/parapet/grafana/dashboards/`; host provisions via Grafana API or file mount | Generated files are host-owned after generation — host can modify |
| OpenTelemetry | Optional; Parapet emits OTel span attributes for SLO events when `:opentelemetry` dep present | OTel is an output adapter, not the domain model |
| Sigra (auth) | Optional integration via `Parapet.Integrations.Sigra`; attaches to Sigra telemetry events for login SLO | Compiles to no-op when `:sigra` absent |
| Oban | Optional integration via `Parapet.Surfaces.Oban`; attaches to Oban telemetry events | Compiles to no-op when `:oban` absent; retry-aware semantics |
| LiveDashboard | Optional page injection; `Parapet.LiveDashboard.Page` implements `Phoenix.LiveDashboard.PageBuilder` | Deferred post-v0.1 unless a single doctor/health page is trivially achievable |

### Internal Boundaries

| Boundary | Communication | Notes |
|----------|---------------|-------|
| `Parapet.Telemetry` ↔ `Parapet.Metrics` | Direct function calls; telemetry calls `Metrics.emit/2` | No GenServer message-passing needed on the hot path |
| `Parapet.Surfaces.*` ↔ `Parapet.Telemetry` | Surfaces register handler specs; Telemetry owns attach lifecycle | Surfaces declare event names; Telemetry wires them up |
| `Parapet.SLO` ↔ `Parapet.Dashboards` | SLO structs passed to dashboard generators at compile time | Dashboard generation is a pure function of SLO definitions |
| Host `MyApp.Parapet` ↔ `Parapet.Supervisor` | Host module is passed as config to supervisor at startup | `Parapet.Supervisor` reads host module's compiled SLO/surface declarations |
| `Parapet.Doctor` ↔ all modules | Doctor reads compiled config from host module; no runtime coupling | Doctor is a Mix task, not an OTP process |
