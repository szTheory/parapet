# V1 Design Fork: Demo App Shape (DEMO-01)

**Question:** What is the right shape for Parapet's runnable demo/example app, and how do we keep it CI-green without bloating the Hex package?

**Researched:** 2026-05-25
**Overall confidence:** HIGH (ecosystem patterns verified from source; Parapet codebase read directly)

---

## 1. Decision Question

DEMO-01 was deferred from v0.10 specifically "to validate that docs alone reduce onboarding friction first." Now that docs are proven (7 guides, <30-min getting-started), the demo's job is **not to duplicate the guide** — it is to make the guide *runnable end-to-end* and to give a prospective adopter a live, populated Operator UI with real-looking incidents, SLO state, and runbook steps. It is also the proof surface that `mix parapet.install` produces a working host app and that the CI lane stays green.

The decision: **what shape does DEMO-01 take?**

Constraints from first principles:
- Parapet's `files:` whitelist (`lib priv .formatter.exs mix.exs README* CHANGELOG* LICENSE* docs`) must never include the demo.
- The demo must be CI-compiled and exercised; "CI green check" is the explicit DEMO-01 requirement.
- Parapet requires Postgres (Ecto-backed incidents, timeline entries, tool audits). No SQLite shortcut.
- The demo must exercise `mix parapet.install` (or its Igniter sub-tasks) — the install path is the primary adopter surface.
- Operator UI needs *populated* data to be worth demoing (blank incident list tells adopters nothing).
- Solo maintainer: low ongoing maintenance burden is critical. Demo rot kills credibility faster than no demo.
- The demo must NOT become a documentation footnote; it must be first-class in the README and getting-started guide.

## 2. Options Considered

### Option A: Full child Phoenix app under `examples/demo_app/`

**Description:** A standalone `mix new --sup` / `phx.new` project living at `examples/demo_app/` with its own `mix.exs` that path-depends on parapet via `{:parapet, path: "../.."}`. Has its own `config/`, `priv/repo/seeds.exs`, `priv/repo/migrations/`, and a `mix setup` alias that chains `deps.get + ecto.setup + run seeds`.

**Concrete structure:**
```
examples/demo_app/
  mix.exs              # {:parapet, path: "../.."}; {:phoenix, ...}; {:oban, ...}; {:faker, ...}
  mix.lock
  config/
    config.exs         # parapet providers: [Parapet.SLO.StarterPack.WebSaaS]
    dev.exs
    test.exs
  lib/
    demo_app/
      application.ex
      parapet_instrumenter.ex   # generated by mix parapet.install
    demo_app_web/
      endpoint.ex               # has Parapet.Plug.Metrics
      router.ex                 # mounts Parapet Operator UI
  priv/
    repo/
      migrations/               # parapet spine migrations (copied from generator output)
      seeds.exs                 # 5 seeded incidents in various states; 2 runbooks; timeline entries
  test/
    demo_app_web/
      operator_smoke_test.exs   # GET /parapet → 200; incident list non-empty
```

CI job: a separate `demo` job in `ci.yml` that:
```yaml
- run: cd examples/demo_app && mix deps.get
- run: cd examples/demo_app && mix ecto.create && mix ecto.migrate
- run: cd examples/demo_app && mix run priv/repo/seeds.exs
- run: cd examples/demo_app && mix test
```

Hex exclusion: automatic — `examples/` is not in Parapet's `files:` whitelist. Zero risk of shipping to Hex.

**Pros:**
- Closest to a real host app. The demo IS what an adopter's app will look like after install.
- Path dep means demo always tracks parapet HEAD with zero version drift.
- `mix setup` + `mix phx.server` is a single, self-documenting DX (standard Phoenix onboarding UX).
- Seed data can be rich: realistic incidents with timeline entries, tool audits, SLO state.
- Can house a smoke test (`mix test`) that asserts Operator UI returns 200, incident list is non-empty.
- Credo, format, and dialyzer exclusions from root project are clean — the demo has its own quality gates.
- No dev-only deps leak into Parapet's main `mix.exs`.
- Precedent: PromEx `example_applications/web_app/` uses exactly this shape (path dep `{:prom_ex, path: "../../"}`), full mix.exs, Postgres, Oban.

**Cons:**
- Separate `mix.lock` to maintain (Dependabot can be configured; low burden in practice).
- CI job adds ~3-4 min (Postgres service already running for main test job; can reuse).
- If parapet's public API changes, the demo's code may need updating (this is a feature, not a bug — forces the maintainer to keep the demo honest).
- Requires Phoenix as a demo-only dep (already an implicit transitive dep; explicit is fine).

**Tradeoffs:**
- Highest fidelity. Highest long-term maintenance honesty (breaking changes surface here). Best for onboarding.

### Option B: `dev.exs` + dev-only routes (à la Phoenix LiveDashboard / ErrorTracker)

**Description:** A single `dev.exs` file at the Parapet repo root that uses `Mix.install/2` + `PhoenixPlayground.start/1` to spin up a minimal Phoenix endpoint demonstrating Parapet's Operator UI. Dev-only deps (`phoenix_playground`, `circular_buffer`, etc.) are in Parapet's `mix.exs` under `only: :dev`.

**Concrete structure:**
```elixir
# dev.exs (root of parapet repo)
Mix.install([
  {:parapet, path: "."},
  {:phoenix_playground, "~> 0.1"},
  {:faker, "~> 0.17", only: :dev},
  ...
])

Application.put_env(:parapet, :repo, DevRepo)
# ... define DevRepo, seed inline, start supervisor
PhoenixPlayground.start(router: DevRouter)
```

**Pros:**
- Zero additional project to maintain (no separate `mix.lock`).
- ErrorTracker uses exactly this pattern — iex dev.exs — and it works.
- All code lives in one file; easy to grep, diff, and review.
- dev-only deps stay visibly separate from runtime.

**Cons:**
- `Mix.install/2` is designed for script files, not for a project with existing deps. In a mix project context, using `Mix.install` inside `dev.exs` when you already have a `mix.exs` is awkward (Mix.install raises if Mix is already started with a project).
- Phoenix LiveDashboard solves this differently: dev.exs is NOT a Mix.install script; it is run as `mix run dev.exs` and adds dev-only modules inline, using the project's existing deps. This is cleaner but requires dev deps to live in Parapet's `mix.exs`.
- **Postgres problem:** PhoenixPlayground does not support Ecto/Postgres. ErrorTracker uses SQLite via `ecto_sqlite3` because its dev.exs is demonstrating error capture, not Ecto-heavy evidence. Parapet's core value (incident spine, timeline entries, tool audits) is intrinsically Postgres-backed. Faking it with SQLite loses the entire evidence story.
- No `mix parapet.install` exercise: the dev.exs is hand-wired, not generator-produced. The install path — the primary adopter experience — is never tested.
- Seed data is awkward: defining 15+ Ecto records inline in a dev.exs is messy and hard to maintain.
- "Run `iex dev.exs`" is less discoverable than "run `mix setup && mix phx.server`".
- Dev deps pollute Parapet's `mix.exs` and could confuse adopters scanning dependencies.

**Tradeoffs:**
- Works well for libraries without Ecto dependencies (ErrorTracker's SQLite pivot) or libraries with simple UI. Not a fit for Parapet's Postgres-first evidence model.

### Option C: Livebook-driven demo

**Description:** A `.livemd` notebook under `livebook/parapet_demo.livemd` that walks through the Parapet concepts interactively, using `Mix.install` to pull parapet from Hex (or path), connecting to a local Postgres, and rendering tables of incidents and SLO state.

**Concrete structure:**
```
livebook/
  parapet_demo.livemd
  README.md
```

**Pros:**
- No CI dependency (notebooks are documentation, not executable CI).
- Appealing for educators and conference demo scenarios.
- Livebook's Kino framework can render incident tables, SLO states interactively.

**Cons:**
- Livebook notebooks cannot be CI-compiled in a meaningful way on GitHub Actions without a running Livebook server (the `livebook server` CLI exists but adds significant CI complexity and latency).
- DEMO-01's requirement is explicitly a **CI green check** — notebooks do not satisfy this.
- Requires adopters to install Livebook separately.
- `Mix.install` in notebooks pins specific Hex versions; a path dep works but is clunky.
- Notebooks go stale fastest of all artifacts — no compile-time enforcement, no type checking.
- The getting-started guide flow (`mix parapet.install` → `mix parapet.gen.prometheus` → `mix parapet.doctor`) is fundamentally a command-line story, not a notebook story.

**Verdict:** Ruled out as the primary demo. Could be a bonus `livebook/` artifact in a future milestone.

---

### Option D: Single-file Plug/Phoenix (Bandit + Phoenix.Endpoint in one script)

**Description:** A single `examples/demo.exs` (or `demo.ex`) that starts Bandit + Phoenix.Endpoint + Ecto.Repo inline, defining everything in one file, using `elixir demo.exs` or `mix run examples/demo.exs`.

**Concrete structure:**
```elixir
# examples/demo.exs
Application.ensure_all_started(:parapet)
# define inline repo, router, seed data in-process
```

**Pros:**
- Minimal files.
- Could in theory be run without a full Phoenix install.

**Cons:**
- Parapet's Igniter-based `mix parapet.install` cannot be exercised — it requires a project context (a host `mix.exs`) to compose tasks.
- Defining migrations + Ecto repo + Phoenix endpoint + router in one file becomes a 300-line mess.
- No standard test harness.
- "Run `elixir examples/demo.exs`" requires careful environment setup and is not a DX pattern adopters will recognize.
- Harder to keep formatted, linted, and CI-green than a proper mix project.

**Verdict:** Only suitable for illustrative scripts. Not a fit for DEMO-01's CI green check requirement.

## 3. What's Idiomatic in Elixir/Phoenix/Ecto

**Verified patterns from source:**

| Pattern | Used by | Structure | Postgres | Exercises installer? | CI-tested? |
|---------|---------|-----------|----------|---------------------|------------|
| Full child app (Option A) | PromEx | `example_applications/web_app/`, own `mix.exs`, path dep | YES (Postgrex) | Can be | YES |
| `dev.exs` inline + Mix.install | ErrorTracker | `dev.exs`, PhoenixPlayground, SQLite | NO (SQLite) | NO | NO |
| `dev.exs` run in project | LiveDashboard | `dev.exs`, project deps only, `mix run dev.exs` | N/A (no Ecto) | NO | NO |
| Test host app (`test/example_app/`) | WyeWorks pattern | `test/example_app/`, path dep, `mix e2e` alias | YES possible | Can be | YES |
| Livebook | N/A | `.livemd` | Possible | NO | NO |

**Key observation:** The only idiomatic pattern that satisfies all three constraints simultaneously — (1) Postgres/Ecto, (2) exercises installer, (3) CI-green check — is the **full child Phoenix app** placed either under `examples/` or `test/example_app/`.

The `dev.exs` pattern is idiomatic for libraries whose demo has no Ecto requirements (LiveDashboard, ErrorTracker w/ SQLite). For Parapet, the evidence spine is the core product. A demo without real Postgres incidents is a lie.

**`test/example_app/` vs `examples/demo_app/`:**
Both are equally excluded from Hex `files:`. The WyeWorks pattern puts it in `test/` so a single `mix test --include integration` from the root can discover it without a `cd`. The PromEx pattern puts it in `examples/` for clarity of purpose. For Parapet, `examples/demo_app/` is preferred because:
- The demo is a *consumer* artifact (not a test fixture) — it should feel like a real app, not a test helper.
- It is linked from the README as "clone and run."
- Separating it from `test/` avoids confusion with the library's own test suite.
- The CI job for the demo is a dedicated lane, not mixed into `mix test`.

## 4. Lessons From Comparable Libraries

### Phoenix LiveDashboard — DONE RIGHT
- `dev.exs` at repo root; run with `mix run dev.exs`. All dev deps under `only: :dev` in `mix.exs`.
- No dev.exs in `files:` whitelist; circular_buffer, telemetry_poller listed as dev-only.
- Provides rich fake telemetry (History GenServer) so the UI is populated immediately.
- **Right:** CI does NOT run the dev.exs. It is purely a local development aid. LiveDashboard has no Ecto/Postgres requirement, so SQLite or in-memory is irrelevant.
- **Lesson for Parapet:** dev.exs works when your demo has no DB. Parapet's DB requirement makes this pattern insufficient.

### ErrorTracker — DONE RIGHT (for its context)
- `iex dev.exs` with PhoenixPlayground + SQLite via `ecto_sqlite3`. No seed data — demonstrates via intentional crashes in the UI.
- **Right:** Clever use of SQLite to avoid the Postgres dependency. Interactive crash buttons are great UX.
- **Wrong:** CI-tested? No. Demo cannot be CI-green-checked.
- **Footgun avoided:** Did not try to fake Postgres; matched the demo's scope to what the library actually requires (SQLite is legitimate for error tracking, not for Parapet's incident evidence model).
- **Lesson for Parapet:** If your core value lives in Postgres rows, SQLite won't validate it.

### PromEx — DONE RIGHT
- `example_applications/web_app/` with own `mix.exs`, path dep `{:prom_ex, path: "../../"}`, Postgrex, Oban, Broadway.
- Full Phoenix app. Excluded from Hex `files:` by not being in the whitelist.
- **Right:** Demonstrates a real-world integration scenario with multiple dependencies, Postgres, real app structure.
- **Wrong (reported by community):** Demo became stale vs. PromEx main when Phoenix version was bumped in the example but not pinned consistently. Lock file drift caused periodic CI failures.
- **Footgun:** Separate `mix.lock` in the child app. Fix: Dependabot configured for the `examples/` directory, and CI fails loudly on `mix deps.unlock --check-unused`.
- **Lesson for Parapet:** Path dep is the right call. Lock file needs Dependabot attention. Pin the Phoenix version conservatively.

### Oban — LESSON
- Oban's own test suite runs against Postgres. There is no separate demo app. Oban Web (commercial) has a live demo site.
- **Right:** Oban's main test suite IS the CI proof. For open-source libraries that are stateless enough, this is sufficient.
- **Lesson for Parapet:** Parapet's library test suite already validates the library. The demo's job is specifically the adopter onboarding proof — the `mix parapet.install` golden path exercise. They are different proofs.

### Bandit — LESSON
- Bandit uses a `dev.exs` pattern for local development; no dedicated demo app.
- **Lesson for Parapet:** Fine for pure protocol libraries. Not applicable.

### Ash Framework — LESSON
- Ash uses dedicated demo apps (AshHQ, the Ash Realworld app) as separate repos. They are not embedded in the library repo.
- **Right:** Separation keeps demo quality high and library CI fast.
- **Wrong:** Separate repos drift faster. A path dep in the same repo is strongly preferable for a solo maintainer.
- **Footgun:** Separate-repo demos go stale in weeks without automation. Same-repo path dep demos go stale in months.

### Phoenix LiveBeats — LESSON
- LiveBeats is a standalone app demonstrating LiveView. Not a library demo.
- **Lesson for Parapet:** LiveBeats' seed data approach (users, songs, follows in `priv/repo/seeds.exs`) is the right model. Realistic data makes the UI feel alive.

### Stale Demo Anti-Pattern (Cross-ecosystem)
- The most common failure mode for library example apps: they compile on day of creation, then drift as the library evolves, then break silently because the CI job was "optional" or path-gated behind a flag.
- **Prevention:** The demo CI job must be in `release_gate` (required), never optional. No `continue-on-error: true`. The maintainer must feel the pain of demo breakage before merging.

## 5. DX/UX Considerations (Principle of Least Surprise)

### Onboarding DX target: zero-to-populated-UI in one terminal session
The adopter who discovers the demo repo should be able to run:
```bash
git clone https://github.com/szTheory/parapet
cd parapet/examples/demo_app
mix setup          # deps.get + ecto.create + ecto.migrate + run seeds
mix phx.server     # opens http://localhost:4000/parapet
```
and see a running Operator UI with seeded incidents, timeline entries, and SLO state within ~2 minutes.

This matches Phoenix's own `mix setup` convention (defined in `aliases` in `mix.exs`) and is the single most recognizable DX in the Elixir ecosystem.

### What the seeded UI must show
For the demo to prove Parapet's value, the seeded state must include:
1. **At least 3 incidents in different states**: one `open` (firing), one `investigating` (acknowledged), one `resolved` (with retrospective).
2. **Timeline entries on each incident**: at least a `note` entry and an `alert_received` entry per incident to show the evidence spine.
3. **A runbook step**: the open incident should have a runbook attached, with one `warning:` step visible.
4. **One tool audit record**: to show the ToolAudit surface.
5. **SLO state visible**: at least one SLO slice registered via `Parapet.SLO.StarterPack.WebSaaS` so `mix parapet.doctor` passes.

These are inserted by `priv/repo/seeds.exs` using `Ecto.Repo` directly. No `ex_machina` or `faker` needed for seed data this size — plain Ecto changesets are more readable and have no extra dependency.

### Principle of least surprise checklist
- `mix setup` is the entry point — every Phoenix developer knows it.
- `mix phx.server` starts the app — no novel commands.
- The demo README links to each step in the getting-started guide, so the demo doubles as an interactive walkthrough.
- `http://localhost:4000/parapet` is the Operator UI; `/metrics` is the Prometheus scrape endpoint.
- If Postgres is not running, the error message is clear: standard Ecto "connection refused" — no novel error.
- Docker Compose is optional convenience, not required. Bare Postgres (e.g. `brew services start postgresql`) is the default.

### What NOT to include in the demo
- No custom auth (mounts Operator UI without authentication — for demo only, README prominently notes this).
- No Oban workers firing jobs (avoids timer noise in the demo).
- No external integrations (Sigra, Mailglass) — keep the demo to the pure Parapet core.
- No Docker Compose in v1.0 — it adds CI complexity and "Docker is required" will block some adopters. Add in v1.1 if requested.

## 6. Recommendation

### The shape: Full child Phoenix app at `examples/demo_app/`

**Exact structure:**

```
examples/
  demo_app/
    mix.exs                         # path dep on parapet; phoenix; oban (optional, for install completeness)
    mix.lock
    .formatter.exs
    config/
      config.exs                    # config :parapet, repo: DemoApp.Repo, providers: [...]
      dev.exs                       # endpoint debug config
      test.exs                      # test repo config
    lib/
      demo_app/
        application.ex              # starts Repo, Endpoint
        parapet_instrumenter.ex     # generated by mix parapet.install (committed as-is)
        runbooks/
          login_failure_runbook.ex  # host-owned runbook example
      demo_app_web/
        endpoint.ex                 # plug Parapet.Plug.Metrics wired in
        router.ex                   # scope "/parapet" → ParapetOperatorLive
    priv/
      repo/
        migrations/                 # parapet spine migrations (output of mix parapet.gen.spine)
        seeds.exs                   # 5 realistic incidents + timelines + SLO state
    test/
      demo_smoke_test.exs           # GET /parapet → 200; assert incident count > 0
      test_helper.exs
    README.md                       # "Clone this repo, run mix setup, visit /parapet"
```

**Key design decisions:**

1. **Path dep, not Hex dep.** `{:parapet, path: "../.."}` ensures the demo always tracks the library HEAD. This is what PromEx does; it is correct.

2. **Committed generator output.** The files `parapet_instrumenter.ex` and the migrations under `priv/repo/migrations/` are the actual output of `mix parapet.install` run against the demo app. They are committed. This proves: (a) the generator works, (b) what a real host app looks like after install. CI does NOT re-run `mix parapet.install` (that would be a golden-diff test, which is a separate concern). The demo shows the *result* of a correct install.

3. **Seed data via `priv/repo/seeds.exs`.** Uses plain `DemoApp.Repo.insert!/1` calls with Parapet schemas directly. No ex_machina, no faker. The seed creates: 3 incidents (open/investigating/resolved), 2 timeline entries per incident, 1 tool audit record, 1 runbook. This is ~60 lines of Elixir and trivially readable.

4. **Smoke test.** `test/demo_smoke_test.exs` makes a live HTTP request to `GET /parapet` via `Plug.Test` or `Phoenix.ConnTest` and asserts `200`. It also calls `DemoApp.Repo.aggregate(Parapet.Ecto.Incident, :count)` and asserts `> 0`. This is the "CI green check" for DEMO-01.

5. **`mix setup` alias.**
```elixir
# examples/demo_app/mix.exs
defp aliases do
  [
    setup: ["deps.get", "ecto.create", "ecto.migrate", "run priv/repo/seeds.exs"]
  ]
end
```

6. **Getting-started guide integration.** The guide's final "Next steps" section gains a new bullet: "Run the demo app: `cd examples/demo_app && mix setup && mix phx.server`". The demo's README links back to the guide for each step. The two artifacts cross-reference each other.

### How it's CI-kept-green

Add a `demo` job to `.github/workflows/ci.yml`:

```yaml
demo:
  runs-on: ubuntu-latest
  needs: [test]   # only run if main test suite passes
  services:
    postgres:
      image: postgres:15-alpine
      env:
        POSTGRES_USER: postgres
        POSTGRES_PASSWORD: postgres
        POSTGRES_DB: parapet_demo_test
      ports: ['5432:5432']
      options: >-
        --health-cmd pg_isready
        --health-interval 10s
        --health-timeout 5s
        --health-retries 5
  env:
    MIX_ENV: test
    PGUSER: postgres
    PGPASSWORD: postgres
    PGHOST: localhost
  steps:
    - uses: actions/checkout@v4
    - uses: erlef/setup-beam@v1
      with:
        elixir-version: '1.19.0'
        otp-version: '27.2'
    - name: Cache demo deps
      uses: actions/cache@v4
      with:
        path: examples/demo_app/deps
        key: ${{ runner.os }}-demo-mix-${{ hashFiles('examples/demo_app/mix.lock') }}
    - name: Install demo deps
      run: cd examples/demo_app && mix deps.get
    - name: Create and migrate demo DB
      run: cd examples/demo_app && mix ecto.create && mix ecto.migrate
    - name: Run demo seeds
      run: cd examples/demo_app && mix run priv/repo/seeds.exs
    - name: Demo smoke test
      run: cd examples/demo_app && mix test
```

Add `demo` to `release_gate`:
```yaml
release_gate:
  needs: [lint, test, demo]
  ...
```

The `demo` job reuses the Postgres service container pattern already in the rulestead CI template (sibling lib DNA). No new infrastructure.

### How it's excluded from Hex

`examples/` is simply not in Parapet's `files:` whitelist:
```elixir
files: ~w(lib priv .formatter.exs mix.exs README* CHANGELOG* LICENSE* docs)
```
No additional action needed. `examples/` is implicitly excluded. This is the standard Hex whitelist mechanism.

Verify it is excluded by running `mix hex.build --dry-run` and confirming `examples/` does not appear in the tarball file list.

### Protecting against demo rot

1. **`demo` is in `release_gate` (required check).** No merge to main unless demo is green.
2. **Dependabot on `examples/demo_app/`.** Add to `dependabot.yml`:
```yaml
- package-ecosystem: "mix"
  directory: "/examples/demo_app"
  schedule: { interval: "weekly" }
```
3. **Path dep means no version drift.** The demo always uses parapet at HEAD; no Hex version pinning to forget to update.
4. **Smoke test is narrow.** `GET /parapet → 200` and `incident count > 0` are stable assertions. They will not break unless the Operator UI is removed or the Ecto schema changes fundamentally — both of which are correct failures.

## 7. Coherence With Parapet's Vision & Other v1.0 Decisions

**Host-owned and inspectable:** The demo is a complete, committed host app — not a generated scaffold. Adopters read `parapet_instrumenter.ex` and see exactly what `mix parapet.install` produces. This is the "host-owned" principle embodied.

**API / telemetry freeze:** The v1.0 freeze decision means the telemetry contracts and recovery SLO surfaces will be stable. The demo app crystallizes those stable surfaces into a concrete example. A demo built before the freeze forces the freeze to be real (any change to the public surface breaks the demo and the CI gate). This is the right pressure.

**Happy path short:** The getting-started guide's 5-step happy path (`add dep → install → activate WebSaaS → gen.prometheus → doctor`) is the demo's `mix setup`. The demo makes the guide *runnable* without duplicating it.

**Engineering DNA (example hosts, golden installer tests):** The DNA doc explicitly calls for "example hosts" and "golden installer tests" once the install surface stabilizes. The demo app IS the example host. The committed generator output IS the golden installer evidence (the demo's committed files serve as the expected output of a successful install). If the installer changes such that the demo's files drift from what the generator would produce, a CI diff check can be added in v1.1 (golden-diff test). That is out of scope for v1.0 DEMO-01.

**No Hex shipping:** The `files:` whitelist is already in place. `examples/` is already absent. No change needed.

**Coherence with deferred items:**
- `SLO-W1` (interactive wizard): the demo app can be updated to use the wizard output when that ships.
- `SLO-B1` (cross-integration bundles): demo can add a `--with-sigra` variant in v1.1.
- The demo does NOT include Livebook, Docker Compose, or Playwright — those are v1.1+ concerns.

## 8. Milestone Fit (v1.0) + Rough Effort + Suggested GSD Phase Chunking

**Milestone fit: v1.0.** DEMO-01 is the explicitly named add-on for v1.0. The milestone deliverable is "a runnable demo app + CI green check." This research validates that the scope is achievable and well-bounded.

**Rough effort (solo maintainer):**

| Work item | Rough effort |
|-----------|-------------|
| `examples/demo_app/` scaffolding (`mix new`, `mix.exs`, configs) | 1-2h |
| Wire Parapet: copy generator output (instrumenter, endpoint plug, spine migrations) | 1h |
| `priv/repo/seeds.exs` with 3 incidents + timelines + tool audit | 1-2h |
| Smoke test (`demo_smoke_test.exs`) | 30min |
| CI `demo` job in `ci.yml` + `release_gate` update | 30min |
| Demo `README.md` + getting-started guide link | 30min |
| Dependabot `examples/demo_app` entry | 10min |
| Verify `mix hex.build --dry-run` excludes `examples/` | 10min |
| **Total** | **~5-7h** |

This is a single, focused milestone phase — no need to split further for v1.0. The work is sequential (scaffold → wire → seed → test → CI → docs) but each step is fast.

**Suggested GSD phase structure for DEMO-01:**

**Phase 1 (DEMO-APP-CORE):** Scaffold `examples/demo_app/`, wire parapet path dep, commit generator output (instrumenter + migrations), basic application supervision tree. Goal: `mix ecto.create && mix ecto.migrate` passes.

**Phase 2 (DEMO-SEED-TEST):** Implement `priv/repo/seeds.exs` (5 incidents, timelines, SLO state), write `demo_smoke_test.exs`, add `mix setup` alias. Goal: `mix setup && mix test` is green locally.
