# Requirements: Parapet

**Defined:** 2026-05-09
**Core Value:** A Phoenix SaaS team can install Parapet and immediately know whether their critical user journeys are healthy — with evidence, not just dashboards.

## v0.1 Requirements

Requirements for the "Trustworthy spine" initial release.

### Package and Distribution

- [ ] **PKG-01**: `mix.exs` declares a single `parapet` Hex package with a `files:` whitelist that excludes test fixtures, prompts, and planning documents
- [ ] **PKG-02**: All optional integration dependencies (`:sigra`, `:oban`, etc.) are declared as `optional: true` and compile out cleanly when absent — no `UndefinedFunctionError` or `CompileError` when the dep is missing
- [ ] **PKG-03**: The public module surface is explicitly delimited — only modules in `Parapet.*` with `@moduledoc` are part of the public API; internal modules are in `Parapet.Internal.*` or marked `@moduledoc false`
- [ ] **PKG-04**: `mix verify.public_api` (or equivalent) produces a machine-readable manifest of the public module/function surface and exits non-zero if undocumented modules are present

### Telemetry Contract

- [ ] **TELE-01**: Every telemetry event emitted by Parapet is documented in `docs/telemetry.md` with its full event name, measurements map, metadata map, and which fields are considered low-cardinality safe
- [ ] **TELE-02**: The telemetry event schema version is declared in the package version and treated as a semver-breaking change if measurement or metadata field names are removed or renamed
- [ ] **TELE-03**: No telemetry event includes high-cardinality fields (user ID, account ID, request ID, session token, raw path) in metric labels — only in structured metadata available to event subscribers
- [ ] **TELE-04**: Telemetry event handlers are attached via `Parapet.attach/1` which returns `{:ok, handlers}` or `{:error, reason}` — never raises on misconfiguration

### HTTP/API Metrics Slice

- [ ] **HTTP-01**: Parapet emits a `parapet.http.request` telemetry event for every completed Phoenix request with measurements `[:duration_ms, :status_code]` and metadata `[:route, :method, :status_class]`
- [ ] **HTTP-02**: The `:route` label uses the Phoenix router's matched route pattern (e.g., `/users/:id`) — never the raw request path — ensuring bounded label cardinality
- [ ] **HTTP-03**: The `:status_class` label is one of `["2xx", "3xx", "4xx", "5xx"]` — never the raw integer status code
- [ ] **HTTP-04**: Parapet exposes a `Parapet.Metrics.HTTP` module that registers a Prometheus counter (`parapet_http_requests_total`) and histogram (`parapet_http_request_duration_ms`) with labels `[route, method, status_class]`
- [ ] **HTTP-05**: Routes with no Phoenix match (404s from unknown paths) are bucketed under the label `route: "_unknown"` rather than the raw path
- [ ] **HTTP-06**: The HTTP plug (`Parapet.Plug.Metrics`) can be added to an existing Phoenix Endpoint without requiring changes to the router and without altering request/response behavior

### Oban/Job Health Slice

- [ ] **OBAN-01**: When `:oban` is present, Parapet emits `parapet.oban.job` telemetry events with measurements `[:duration_ms, :queue_time_ms]` and metadata `[:worker, :queue, :state]` where `:state` is one of `[:success, :failure, :cancelled, :discarded]`
- [ ] **OBAN-02**: A single job exception does not trigger an alert — Prometheus alerting rules generated by Parapet use `rate()` over a minimum window (default 5 minutes) and require a minimum volume threshold before alerting
- [ ] **OBAN-03**: Parapet exposes `Parapet.Metrics.Oban` that registers `parapet_oban_jobs_total` (counter, labels: `[worker, queue, state]`) and `parapet_oban_job_duration_ms` (histogram, labels: `[worker, queue]`)
- [ ] **OBAN-04**: When `:oban` is absent, `Parapet.Metrics.Oban` is not compiled into the application and no runtime errors occur if `Oban` modules are referenced only inside `if Code.ensure_loaded?(Oban)` guards

### Ecto DB Metrics Slice

- [ ] **ECTO-01**: Parapet emits `parapet.ecto.query` telemetry events with measurements `[:total_time_ms, :query_time_ms, :queue_time_ms, :decode_time_ms]` and metadata `[:repo, :source, :result]`
- [ ] **ECTO-02**: `queue_time_ms` and `query_time_ms` are reported as separate Prometheus histograms (`parapet_ecto_queue_time_ms`, `parapet_ecto_query_time_ms`) so operators can distinguish pool saturation from slow queries
- [ ] **ECTO-03**: The `:source` label is the Ecto schema table name or `"_raw"` for raw queries — never parameterized query strings or user data

### SLO DSL

- [ ] **SLO-01**: Adopters define SLOs using `Parapet.SLO.define/2` with a keyword spec containing `objective:` (float 0..1), `window:` (atom like `:30d`), `good_events:` (metric selector), and `total_events:` (metric selector) — the function validates the spec and returns `{:ok, slo}` or `{:error, errors}`
- [ ] **SLO-02**: `Parapet.SLO.to_prometheus_rules/1` takes a compiled SLO struct and returns a string of valid Prometheus YAML containing multi-window burn-rate recording rules and alerting rules — no hand-written PromQL required
- [ ] **SLO-03**: Generated alerting rules include both a fast-burn alert (short window, high burn rate) and a slow-burn alert (long window, low burn rate) per the Google SRE Workbook multi-window pattern
- [ ] **SLO-04**: Generated recording rules use `sum(rate(...))` in the numerator and denominator separately before dividing — never `rate(sum(...))` — to avoid ratio aggregation errors
- [ ] **SLO-05**: Each SLO definition requires a `runbook:` field containing a non-empty string (URL or relative path) — `mix parapet.doctor` fails with exit code 2 if any SLO is missing a runbook

### Login Journey SLO (Sigra Integration)

- [ ] **AUTH-01**: When `:sigra` is present, `Parapet.Integrations.Sigra` subscribes to Sigra's authentication telemetry events and emits `parapet.journey.login` events with measurements `[:duration_ms]` and metadata `[:outcome]` where `:outcome` is `:success` or `:failure`
- [ ] **AUTH-02**: `Parapet.SLO.LoginJourney` ships a pre-built SLO definition for auth success rate with a default 99.5% objective over a 30-day window — adopters can override objective and window
- [ ] **AUTH-03**: `mix parapet.install --with-sigra` generates the Sigra integration config block in the host app's `config/config.exs` and attaches the telemetry handler
- [ ] **AUTH-04**: When `:sigra` is absent, `Parapet.Integrations.Sigra` is not compiled and no runtime warnings or errors appear in application startup logs

### Deploy/Change Markers

- [ ] **DEPL-01**: `Parapet.Deploy.mark/1` accepts a keyword list with `sha:`, `at:` (DateTime), and optional `metadata:` map, emits a `parapet.deploy` telemetry event, and returns `:ok`
- [ ] **DEPL-02**: Deploy markers are exposed as a Prometheus info metric (`parapet_deploy_info`) with labels `[sha, deployer]` that Grafana can use as vertical annotations on SLO panels
- [ ] **DEPL-03**: `mix parapet.install` generates a deploy hook script (`rel/hooks/post_start.sh`) that calls the Parapet deploy marker endpoint — the script is host-owned and modifiable
- [ ] **DEPL-04**: Deploy markers carry a monotonic `sequence` measurement so Grafana queries can order markers correctly even when system clocks drift

### Prometheus Artifacts

- [ ] **PROM-01**: `mix parapet.gen.prometheus` writes valid Prometheus recording and alerting rule YAML files to `priv/parapet/prometheus/` — one file per SLO slice (http, oban, login)
- [ ] **PROM-02**: Generated YAML files pass `promtool check rules` validation without errors
- [ ] **PROM-03**: Generated alerting rules include `annotations.summary` and `annotations.description` fields with human-readable text and a link to the SLO runbook
- [ ] **PROM-04**: Re-running `mix parapet.gen.prometheus` overwrites existing files with a warning if content has diverged from what Parapet would generate — does not silently overwrite adopter customizations without a `--force` flag

### Grafana Artifacts

- [ ] **GRAF-01**: `mix parapet.gen.grafana` writes Grafana dashboard JSON files and a Grafana provisioning YAML to `priv/parapet/grafana/` — importable without manual copy-paste
- [ ] **GRAF-02**: Generated dashboards include panels for: HTTP request rate, HTTP error rate, HTTP p50/p95/p99 latency, Oban job success/failure rate, Oban queue depth, login SLO burn rate, and error budget remaining
- [ ] **GRAF-03**: Grafana dashboards reference only Prometheus recording rules generated by Parapet — no raw PromQL in panel queries that could silently break if rule names change
- [ ] **GRAF-04**: Deploy marker annotations are rendered as vertical lines on all SLO time-series panels

### `mix parapet.doctor`

- [ ] **DOCT-01**: `mix parapet.doctor` exits with code 0 (all clear), 1 (warnings but no hard failures), or 2 (hard safety violation) — codes are stable and documented for use as a CI gate
- [ ] **DOCT-02**: Doctor checks include: (a) `/metrics` endpoint is not publicly accessible without authentication, (b) LiveDashboard is not accessible in production without authentication, (c) no metric label contains a known high-cardinality pattern (raw path, user ID format, UUID format), (d) all defined SLOs have non-empty runbook fields
- [ ] **DOCT-03**: Each doctor check prints a one-line result (`[ok]`, `[warn]`, or `[fail]`) with a short explanation and a link to the relevant section of the Parapet install guide
- [ ] **DOCT-04**: Doctor checks run in under 5 seconds on a standard Phoenix application with no external network calls required
- [ ] **DOCT-05**: `mix parapet.doctor --ci` suppresses color output and emits structured JSON to stdout so CI systems can parse results programmatically

### Install Generator

- [x] **INST-01**: `mix parapet.install` generates a `Parapet.Instrumenter` module in the host app (e.g., `lib/my_app/parapet_instrumenter.ex`) that the adopter can inspect, modify, and own
- [x] **INST-02**: `mix parapet.install` appends the required `Parapet.Plug.Metrics` to the host app's Endpoint in `lib/my_app_web/endpoint.ex` and prints a diff of the change before applying it — requires `--yes` to apply without prompt
- [x] **INST-03**: `mix parapet.install` adds Parapet configuration to `config/config.exs` with inline comments explaining each option
- [x] **INST-04**: `mix parapet.install` is idempotent — running it twice on an already-configured app prints "already configured" for each step and makes no changes
- [x] **INST-05**: `mix parapet.install --dry-run` prints all proposed changes without writing any files

### Documentation

- [ ] **DOCS-01**: `README.md` covers the full path from `mix.exs` dependency addition through: Endpoint plug insertion, first SLO definition, `mix parapet.gen.prometheus`, `mix parapet.gen.grafana`, and verifying the first Grafana panel shows data
- [ ] **DOCS-02**: `docs/telemetry.md` documents every telemetry event with its full name, measurements, metadata, cardinality guidance, and the semver guarantee
- [x] **DOCS-03**: `docs/slo-reference.md` documents the SLO DSL with a complete example SLO definition, the generated recording rule output, and the generated alerting rule output side-by-side
- [ ] **DOCS-04**: `CHANGELOG.md` is maintained by Release Please from Conventional Commits — no manual changelog edits required for patch/minor/major bumps

### Error Handling and Observability

- [x] **ERR-01**: Telemetry handler exceptions are caught and logged at `:error` level with the event name, exception class, and stacktrace — a bad handler never crashes the calling process
- [ ] **ERR-02**: Metric registration errors (duplicate metric name, incompatible label set) are reported at application start with a clear error message identifying the conflicting module — the application does not silently start with a partial metric set
- [ ] **ERR-03**: All `Parapet.*` public functions that can fail return `{:ok, result}` / `{:error, reason}` tagged tuples — no public API function raises for expected error conditions
- [ ] **ERR-04**: Parapet logs at `:debug` when attaching telemetry handlers and at `:warning` when a handler is skipped because its optional dependency is absent

### OSS / CI Discipline

- [ ] **OSS-01**: CI runs `mix format --check-formatted`, `mix credo --strict`, `mix dialyzer`, and `mix test` — all must pass on the main branch
- [ ] **OSS-02**: `mix verify.public_api` is a CI step that fails if a public module lacks `@moduledoc` or a public function lacks `@doc`
- [ ] **OSS-03**: Hex publish uses a `files:` whitelist in `mix.exs` — `prompts/`, `.planning/`, `rel/`, and `*.md` planning files are excluded from the published package
- [ ] **OSS-04**: Release Please is configured to produce GitHub releases and update `CHANGELOG.md` from Conventional Commits on merge to `main`

## v0.2 Requirements

Deferred until the v0.1 telemetry contract is stable and validated in production.

### Durable Evidence Spine

- **EVID-01**: Parapet writes telemetry events to an Ecto-backed `parapet_events` table with columns `[event_name, measurements, metadata, occurred_at]` — schema defined by Parapet migration, owned by host app
- **EVID-02**: Evidence records are queryable via `Parapet.Evidence.query/1` with filters for event name, time range, and metadata fields
- **EVID-03**: The evidence schema is versioned — adding columns is additive; removing or renaming columns is a semver-major change with a migration guide

### Email Deliverability Monitoring

- **EMAIL-01**: When a supported email provider adapter (Postmark, SendGrid, SES) is configured, Parapet subscribes to provider webhook events and emits `parapet.email.delivered`, `parapet.email.bounced`, `parapet.email.complained` telemetry events
- **EMAIL-02**: `Parapet.SLO.EmailDelivery` ships a pre-built SLO for transactional email delivery success rate with configurable objective and window
- **EMAIL-03**: Generated Grafana dashboard includes bounce rate, complaint rate, and delivery latency panels

### Chimeway Notification Health

- **NOTIF-01**: When `:chimeway` is present, `Parapet.Integrations.Chimeway` emits `parapet.notification.sent` and `parapet.notification.failed` events with provider and channel metadata
- **NOTIF-02**: `Parapet.SLO.NotificationDelivery` ships a pre-built SLO for notification delivery success rate

### Wide Events Schema

- **WIDE-01**: `Parapet.Event.emit/2` accepts an event name and a metadata map of arbitrary depth and emits it as a structured log entry via Logger metadata — not as a Prometheus metric label
- **WIDE-02**: High-cardinality fields (`user_id`, `account_id`, `request_id`) are explicitly supported in wide events and explicitly rejected from metric labels — the API surface makes the distinction impossible to confuse

## v0.3+ Requirements

Deferred until product-market fit is established and the evidence model is proven in production.

### In-App Operator UI

- **UI-01**: `parapet_live` package (separate Hex package) provides LiveDashboard-style pages for SLO status, error budget burn, and recent deploy markers
- **UI-02**: UI is read-only — no production mutations from the UI in the first in-app operator surface

### AI Evidence Bundles

- **AI-01**: `mix parapet.gen.agents_md` generates an `AGENTS.md` file containing structured, cited operational context for AI coding assistants — sourced from the durable evidence spine, not hallucinated
- **AI-02**: Each evidence citation in `AGENTS.md` includes a permalink to the specific event record so humans can verify the AI's reasoning

### Approval-Gated MCP Tools

- **MCP-01**: `Parapet.MCP` exposes read-only investigation tools (query evidence, read SLO status) as MCP tool definitions
- **MCP-02**: Production-mutation MCP tools (acknowledge incident, trigger rollback) require an explicit approval step logged to the evidence spine before execution

## Out of Scope

| Feature | Reason |
|---------|--------|
| Hosted SaaS / phone-home telemetry | Violates host-owned principle; creates vendor relationship; changes security boundary |
| APM backend, log database, or trace store | Parapet composes existing systems; rebuilding Prometheus/Loki/Tempo is out of scope |
| Replacement for Phoenix Telemetry, OpenTelemetry, LiveDashboard | Composing these is the point — Parapet is a layer above them |
| Generic cross-language platform | Elixir/Phoenix ecosystem-native is a hard constraint; cross-language is a different product |
| Custom notification/paging system | Teams use PagerDuty/Opsgenie/Slack — generating Prometheus alert rules lets Alertmanager handle delivery |
| Per-user metrics and dashboards | `user_id` in metric labels causes Prometheus cardinality explosion; use wide events instead |
| Autonomous incident remediation | Unbounded production mutations without approval gates cause outages; evidence-first is the model |
| Real-time LiveView incident dashboard (v0.1) | Grafana already serves the operator audience; frontend investment before data model is proven is premature |
| DB-backed durable evidence spine (v0.1) | Commits to a schema before the telemetry contract is stable; deferred to v0.2 |
| Chimeway, Mailglass, Threadline, Rulestead, Accrue, Rindle integrations (v0.1) | Sigra validates the integration pattern; others follow after the spine is proven |
| Synthetic probes / uptime checks | Adds active-probe infrastructure complexity before core passive instrumentation is validated |
