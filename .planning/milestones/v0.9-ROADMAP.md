# Milestone v0.9: Performance, Scale & DX

**Status:** ✅ SHIPPED 2026-05-23
**Phases:** 1-14
**Total Plans:** 36

## Overview

Validate TSDB safety, generator ergonomics, and large-installation behavior. This
milestone shifted focus from feature breadth to operational depth, ensuring Parapet
scales elegantly without bloating the host application's TSDB or Postgres instances.

Phases 1-5 delivered the core operational depth (cardinality protection, database
scale, operator UI performance, unified install, multi-node safety). Phases 6-14
were closure and reconciliation phases that produced milestone-grade verification
surfaces, repaired a regression in the generated operator resolve flow, tightened
archive retention semantics, hardened the multi-node proof for rerunnability, and
backstopped the closure-proof chain so future milestone reruns catch generated
operator UI runtime regressions.

## Milestone Summary

**Decimal Phases:** None. Phases 6-14 were appended (not inserted) as closure and
reconciliation phases after the initial audit surfaced verification and integration gaps.

**Key Decisions:**
- Focus v0.9 on Performance, Scale & DX per the milestone arc — operational depth over feature breadth.
- Unify the installation path under a single `mix parapet.install` Igniter wizard.
- Tackle TSDB safety proactively via `mix parapet.doctor cardinality` + compile-time label limits (max 10/metric).
- Solve Ecto evidence bloat with a built-in `mix parapet.archive` task and an Oban cron template rather than inventing a cold-storage engine.
- Restrict archival to a resolved-only retention contract so active `investigating` work is never pruned.
- Keep the multi-node peer-node canary environment-conditional (skip cleanly without distributed Erlang) rather than failing hard.
- Centralize the repo's recommendation-first, assumptions-mode, low-escalation doctrine in a repo-root `AGENTS.md`.

**Issues Resolved:**
- Generated operator resolve action recorded a note instead of resolving the incident (Phase 13) — repaired and backstopped (Phase 14).
- Archive predicate could prune active work — restricted to resolved-only retention (Phase 10).
- Multi-node proof failed hard with `:nodistribution` in environments without distributed Erlang — made conditional and rerunnable (Phase 11).
- Planning-artifact drift between STATE/ROADMAP/REQUIREMENTS and verified reality — reconciled (Phases 9, 12).
- Public-API documentation gate halted the full suite on new `Parapet.TestSupport.*` modules — gate now excludes the test-support namespace.

**Technical Debt Incurred:**
- Older summary frontmatter records family-level requirement IDs (e.g. `DX-01`, `SCALE-01`) instead of exact sub-IDs, so the three-source requirement cross-check still needs manual interpretation in a few rows.
- Phase 1, 2, and 5 `VALIDATION.md` files exist but do not expose the frontmatter-driven machine classification used by later phases; Phase 6 validation remains a planned-coverage map.
- Phase 4 still depends on an indexed fresh-host adoption transcript rather than a fully automated host-bootstrap lane.
- The advisory benchmark and generated UI proof lane still emit a non-blocking EEx deprecation warning for `<%# ... %>`.
- `.planning/` carries cross-milestone phase-directory contamination (leftover v0.6/v0.7 dirs sharing `01-`/`02-`/`07-` prefixes); retroactive `/gsd:cleanup` recommended.
