## Phase Details

### Phase 1: Durable Evidence Spine (Ecto)
**Goal**: Establish the relational database foundation for modeling incidents, mitigation timelines, and auditable tool executions without storing high-volume telemetry.
**Depends on**: Nothing
**Requirements**: SPINE-01, SPINE-02, SPINE-03, SPINE-04
**Success Criteria**:
  1. Developers can run generators to create Incident, Timeline, and Tool Audit Ecto migrations in their host application.
  2. The system can programmatically transition an incident's state (open, investigating, resolved) and append a durably logged timeline entry for the event.
  3. The system can log an audited tool call (e.g., an AI mitigation action) and link it to a specific incident timeline.
  4. Database performance remains unaffected by telemetry due to explicit separation of low-volume state tracking from high-volume metrics.
**Plans**: 3 plans
- [ ] 01-01-PLAN.md — Create the foundational Ecto schemas
- [ ] 01-02-PLAN.md — Create the Parapet.Evidence context module
- [ ] 01-03-PLAN.md — Create the mix parapet.gen.spine task

### Phase 2: In-App Operator UI (LiveView)
**Goal**: Provide a Phoenix LiveView surface for operators to view incident states and execute verifiable, secure mitigations.
**Depends on**: Phase 1
**Requirements**: UI-01, UI-02, UI-03, UI-04
**Success Criteria**:
  1. Operators can securely view a list of open and historical incidents in a dedicated SRE dashboard.
  2. Operators can view the detailed timeline of events, including audited tool calls, for any selected incident.
  3. Operators can trigger predefined mitigations (e.g., safe mutative actions) and view external runbook or Grafana links from the UI.
  4. The UI components cleanly document their security boundaries, integrating with standard Phoenix authentication out of the box.
**Plans**: TBD
**UI hint**: yes

### Phase 3: Sibling Ecosystem Integrations
**Goal**: Integrate the extended Elixir sibling library ecosystem via compile-time optional adapters to convert sibling state into Parapet SLIs and mitigation actions.
**Depends on**: Phase 1, Phase 2
**Requirements**: ECO-01, ECO-02, ECO-03, ECO-04, ECO-05
**Success Criteria**:
  1. The library compiles and operates correctly when no sibling libraries (Rulestead, Mailglass, etc.) are present in the host application.
  2. When Rulestead is present, operators can correlate feature flag changes with incidents and toggle flags directly via the Operator UI.
  3. When Mailglass or Chimeway is present, the system automatically registers standard deliverability SLIs and triggers timeline events based on delivery failures.
  4. All integrated actions and events generated by sibling adapters are durably recorded in the incident timeline.
