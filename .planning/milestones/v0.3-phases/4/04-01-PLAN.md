# Plan: 04-01-PLAN.md

## Goal
Implement the Incident Acknowledgment workflow, allowing operators to explicitly take ownership of an open incident from the UI.

## Context
Parapet incidents start as `open`. To prevent redundant effort and signal to the team that an incident is being handled, an operator needs to "acknowledge" the incident, which transitions it to `investigating` and logs an immutable timeline entry (`action_type: :acknowledge`). `Parapet.Operator.ActionPayload` already supports `:acknowledge`.

## Tasks

1. **Implement `Parapet.Operator.acknowledge_incident/2`:**
   - Define a new command in `lib/parapet/operator.ex` that takes an incident ID and an `ActionPayload` with `action_type: :acknowledge`.
   - The command should start an Ecto transaction (using `Parapet.Spine.Repo`).
   - Update the `Incident` state to `"investigating"`.
   - Create a `TimelineEntry` marking the acknowledgment (similar to how `execute_mitigation` works).

2. **Extend Operator UI (`parapet.gen.ui`):**
   - Update `priv/templates/parapet.gen.ui/operator_detail_live.ex.eex`.
   - Add an "Acknowledge Incident" button visible only when the incident is in the `"open"` state.
   - Implement the `handle_event("acknowledge", ...)` callback to dispatch the `Parapet.Operator.acknowledge_incident/2` command and show a flash message.

3. **Update Tests:**
   - Add unit tests for `acknowledge_incident/2` in `test/parapet/operator_test.exs`.
   - Ensure the generator tests in `test/mix/tasks/parapet.gen.ui_test.exs` capture the new UI button.

## Verification
- Run `mix test` to confirm all existing and new tests pass.
- Generate the UI in a dummy project and verify the Acknowledge button is visible for open incidents and correctly transitions the state to `investigating`.
