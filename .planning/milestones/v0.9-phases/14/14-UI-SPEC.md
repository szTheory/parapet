## Layout Contract

- Preserve the existing three-pane operator workbench on desktop.
- Left queue pane width: 320px. Contents in order: critical journeys, queue header, active-state scope, refresh banner when present, incident list, cursor pagination, action items.
- Main detail pane width: fluid, target readable measure 720px to 880px. Contents: selected incident summary, proof-relevant current-state facts, canonical timeline.
- Right rail width: 320px. Contents: selected-incident actions only.
- Page max width: 1600px with 32px outer padding on desktop.
- On screens below 960px, keep the existing queue-first and detail-second flow using the same URL-driven state model.
- Do not add tabs, drawers, or a second proof dashboard. The closure proof must remain legible inside the current workbench surface.
- Keep chronology ahead of risky controls: summary first, timeline second, mutation controls after context.

### Closure-Proof Emphasis

- The queue header must keep the `History` entrypoint visible in the first viewport on desktop.
- The active queue and resolved-history surfaces must read as one workflow split across two visible entrypoints, not two unrelated screens.
- The queue-side resolve path must stay visually modest but unmistakable: resolving removes an item from the active queue and makes it reviewable through `History`.

## Component Inventory

- `CriticalJourneysStrip`: compact status strip above the queue; remains visually secondary.
- `IncidentQueueHeader`: queue title, active-only scope label, active count or window summary, and a visible `History` entrypoint.
- `IncidentRefreshBanner`: explicit background-update affordance with calm copy and a single primary button.
- `IncidentQueueList`: bounded current-page list only.
- `IncidentQueueRow`: evidence-first row with state, optional severity, symptom-first title, one bounded secondary line, updated label, and at most one attention chip.
- `IncidentQueuePagination`: previous and next controls plus centered operator-paced label.
- `IncidentSummaryCard`: selected incident title, durable state, impact, and escalation snapshot.
- `IncidentTimeline`: canonical chronology and authoritative proof surface for what happened.
- `ActionRail`: bounded operator actions for the selected incident only.
- `ResolveConfirmation`: destructive confirmation copy that explicitly describes the active-queue to history transition and immutable audit write.

## Interaction Contract

- Queue browsing remains URL-driven through LiveView `handle_params/3`.
- Use hybrid paged cursor navigation. Do not introduce offset pagination, infinite scroll, or silent auto-advance.
- Default queue scope remains active-only: `open` and `investigating`.
- Resolved incidents stay out of the default queue and remain reachable through the explicit `History` entrypoint.
- Default page size remains 30 incidents per page/window.
- Stream only the current page into the DOM with `reset: true` when paging or refreshing.
- Do not silently reorder the visible queue when background updates arrive.
- When background updates exist, show the refresh banner above the queue list with:
  `New incidents or queue changes are available.`
  Primary button label: `Load latest changes`
- Selected incident detail, counters, and badges may update live without changing visible queue order.
- Queue row click selects the incident and preserves queue state in the URL.
- Mobile detail view keeps the explicit `Back to Queue` link; do not replace it with hidden gestures or floating controls.

### Resolve-Flow Contract

- Queue-side `Resolve` must remain a real lifecycle transition through `Parapet.Operator.resolve_incident/2`.
- Successful resolve must remove the incident from the active queue without requiring a full page reload.
- Resolved visibility must remain discoverable through the explicit `History` entrypoint rather than hidden toast-only messaging.
- The UI must never present resolve as note-taking, soft dismiss, or local-only state clearing.
- Any resolve error must preserve the active queue state and selected incident context so the operator can retry or inspect evidence.

### Proof-Surface Discoverability

- Name the closure-proof seam consistently as the `generated resolve-flow proof lane` in operator-facing docs and phase proof surfaces.
- Keep the words `History`, `Resolve`, and `Load latest changes` stable across generator templates, docs, and proof artifacts; avoid synonym drift such as `Archive`, `Dismiss`, or `Sync`.
- The generated UI does not need a visible ‚Äúproof mode,‚Äù but its normal interaction path must make the proof obvious: active queue, resolve action, then resolved history.
- If helper copy is shown near history or empty states, it should reinforce discoverability without teaching a new workflow.

### Queue Row Contract

- Row height target: 72px to 96px depending on whether the secondary line is present.
- Each row must contain exactly these content zones:
  1. State badge.
  2. Severity badge if durable severity exists; otherwise omit.
  3. Symptom-first title.
  4. One compact secondary line from `integration`, `fault_plane`, `affected_journey`, or `queue`.
  5. Compact updated indicator.
  6. At most one attention chip.
- Attention chip vocabulary remains limited to: `Correlated change`, `Approval pending`, `Escalation waiting`.
- Never place proof instructions, runbook prose, toast summaries, or chronology inside the queue row itself.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 | 1.5 |
| Label | 14px | 600 | 1.4 |
| Heading | 20px | 600 | 1.25 |
| Display | 28px | 600 | 1.15 |

### Type Rules

- Use `Display` only for the selected incident title and queue-count headline where present.
- Use `Heading` for pane titles, summary cards, and major empty-state headings.
- Use `Label` for badges, state pills, attention chips, pagination controls, button text, and the `History` entrypoint.
- Relative timestamps and supporting metadata stay at body size with muted color.
- Do not introduce a fifth general-purpose type size for proof affordances; closure-proof emphasis comes from placement and wording, not bigger typography.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#F5F2EA` | App background, main reading surfaces, neutral queue canvas |
| Secondary (30%) | `#E7E0D3` | Queue pane, right rail, cards, banner background, grouped status areas |
| Accent (10%) | `#0F766E` | Selected queue row, refresh CTA, focus ring, active-scope chip |
| Destructive | `#B42318` | Resolve confirmations and destructive actions only |

Accent reserved for: selected queue row treatment, `Load latest changes`, active queue scope chip, keyboard focus ring, current pagination control. Do not promote `History` to an accent button and do not tint resolved-history browsing as a celebratory success state.

### Color Rules

- Reuse the approved Phase 3 palette unchanged for continuity and maintainability.
- State remains the first semantic color signal; use restrained tints, not saturated panels.
- Refresh banner should feel informational and operator-paced, not urgent-alert red.
- `History` stays a quiet text-link affordance with underline by default.
- Resolved-state visuals should remain subdued and evidence-oriented, not ‚Äúall clear‚Äù marketing green.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Load latest changes |
| Empty state heading | No active incidents |
| Empty state body | Open and investigating incidents will appear here. Use History to review resolved incidents without disrupting the active queue. |
| Error state | We couldn't load the incident queue. Refresh the queue. If the problem persists, check the host app database connection and Parapet operator query telemetry, then try again. |
| Destructive confirmation | Resolve incident: Resolve this incident? Confirm only after user impact has stopped and the timeline tells the full story. This removes it from the active queue, makes it reviewable in History, and writes an immutable audit entry. |

### Tone Rules

- Use calm, declarative language.
- Prefer operator-paced verbs such as `review`, `load`, `resolve`, and `confirm`.
- Keep closure-proof wording factual. Do not say the UI ‚Äúproves‚Äù anything to the operator in-product.
- Never imply autonomous action by the UI.
- Avoid congratulatory, theatrical, or dashboard-marketing copy.

---

## State Contract

- Queue loading state uses 6 skeleton rows in the queue pane; do not block the entire workbench after first mount.
- Empty state appears only when the active queue truly has zero rows.
- The active-queue empty state must still leave `History` visibly reachable.
- Refresh banner appears only when background data changed relative to the visible queue snapshot.
- Refresh banner must not cover queue rows and must collapse after the explicit refresh action.
- Queue pagination controls stay visible at the bottom of the queue pane when more rows exist in either direction.
- When no incident is selected, the detail pane uses the neutral empty copy `Select an incident to review details`.
- Selection state must remain visible with both tinted background and persistent accent border.
- Resolve success should not require an additional celebratory banner; the source of truth is the queue transition and resolved-history availability.
- Resolve failure should use concise inline or flash copy: `Failed to resolve`.

---

## Accessibility And Motion

- All queue rows, `History` links, refresh controls, resolve controls, and pagination controls must be keyboard reachable.
- Selected queue row must expose selected-state semantics such as `aria-current="true"` or equivalent.
- Refresh banner button text must remain explicit out of context; do not shorten it to `Refresh`.
- Destructive confirmation copy must name the outcome in plain language, including movement out of the active queue.
- Focus ring color matches accent and must remain visible against dominant and secondary surfaces.
- Use only two motion patterns: 140ms fade for queue refresh and 160ms slide/fade for mobile route transitions.
- Disable non-essential motion when the user prefers reduced motion.
