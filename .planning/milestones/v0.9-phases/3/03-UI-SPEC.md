# Phase 3 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none; use host-owned Phoenix LiveView function components |
| Icon library | Heroicons outline, 20px and 24px only |

### Source Notes

- `Tool: none` is locked by project shape: Elixir/Phoenix generated UI, not React/Next/Vite.
- No `components.json`, Tailwind config, or packaged design system exists in the repo root as of 2026-05-20.
- The generated Operator UI remains host-owned, evidence-first, and inspectable; do not introduce opaque runtime UI abstractions.

---

## Layout Contract

- Preserve the three-pane operator workbench on desktop.
- Left queue pane width: 320px. Contents in order: critical journeys, queue heading, active-state scope, refresh banner when needed, incident list, cursor pagination, action items.
- Main detail pane width: fluid, target readable measure 720px to 880px. Contents: selected incident summary, detail facts, timeline.
- Right rail width: 320px. Contents: action rail only for the selected incident.
- Page max width: 1600px with 32px outer padding on desktop.
- On screens below 960px, collapse to queue-first and detail-second views using the same URL-driven state model.
- Do not add tabs inside the queue pane. Active queue, history access, and refresh state must remain visible in one vertical scan.
- Queue list remains the primary visual locus on first load when no incident is selected. When an incident is selected, the detail summary becomes the visual focal point.

---

## Component Inventory

- `CriticalJourneysStrip`: compact status strip above the queue; keep visually secondary to the queue itself.
- `IncidentQueueHeader`: queue title, active-only scope label, active count, and history entrypoint.
- `IncidentRefreshBanner`: explicit background-update affordance with count-aware copy and a single primary button.
- `IncidentQueueList`: stream-backed list that renders the current page only.
- `IncidentQueueRow`: medium-density evidence-first row with bounded triage facts only.
- `IncidentQueuePagination`: previous and next page controls plus current window summary.
- `IncidentSummaryCard`: selected incident title, state, severity, impact, and escalation snapshot.
- `IncidentTimeline`: canonical chronology; remains visually and semantically separate from queue browsing.
- `ActionRail`: existing bounded operator actions for the selected incident only.

---

## Interaction Contract

- Queue browsing is URL-driven through LiveView `handle_params/3`. Cursor direction, selected incident, and history entrypoint must be inspectable in the URL.
- Use hybrid paged cursor navigation, not offset pagination and not infinite scroll.
- Default queue scope is active-only: `open` and `investigating`.
- Resolved incidents are not shown inline in the default queue. Expose them through a separate history view or explicit history filter entrypoint.
- Default page size is 30 incidents per page/window.
- Stream only the current page into the DOM with `reset: true` when paging or refreshing.
- Do not silently reorder the visible queue when background updates arrive.
- When background updates exist, show a refresh banner above the queue list. Banner copy:
  `New incidents or queue changes are available.`
  Primary button label: `Load latest changes`
- Selected incident detail, counters, and badges may update live without changing visible queue order.
- Queue row click selects the incident and preserves queue state in the URL.
- Mobile detail view uses an explicit back link to the queue; do not use floating action buttons or hidden swipe-only navigation.

### Queue Row Contract

- Row height target: 72px to 96px depending on whether the secondary line is present.
- Each row must contain exactly these content zones:
  1. State badge.
  2. Severity badge if durable severity exists; otherwise omit.
  3. Symptom-first title.
  4. One compact secondary line from `integration`, `fault_plane`, `affected_journey`, or `queue`.
  5. Compact age or last-updated indicator.
  6. At most one attention chip.
- Attention chip vocabulary is limited to: `Correlated change`, `Approval pending`, `Escalation waiting`.
- Never place impact summary, next safe action, runbook prose, external links, or chronology inside the queue row.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 | 1.5 |
| Label | 14px | 600 | 1.4 |
| Heading | 20px | 600 | 1.25 |
| Display | 28px | 600 | 1.15 |

### Type Rules

- Use `Display` only for the selected incident title and queue-count headline.
- Use `Heading` for pane titles, summary cards, and refresh-banner title text if a title is shown.
- Use `Label` for badges, state pills, attention chips, pagination controls, and button text.
- Relative timestamps and terse metadata may use the body size with muted color; do not introduce a fifth general-purpose type size.
- Incident IDs and cursor-debug metadata, if exposed in a maintainer-only state, may use the host app monospace stack at 14px.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#F5F2EA` | App background, main reading surfaces, neutral queue canvas |
| Secondary (30%) | `#E7E0D3` | Queue pane, right rail, cards, banner background, grouped status areas |
| Accent (10%) | `#0F766E` | Selected queue row, refresh CTA, active-only scope chip, focused pagination control |
| Destructive | `#B42318` | Resolve confirmations and destructive actions only |

Accent reserved for: selected queue row treatment, `Load latest changes`, active queue scope chip, keyboard focus ring, current pagination control. Do not use accent for every link, every state badge, or resolved-state decoration.

### Color Rules

- State remains the first semantic color signal; use restrained tints, not full-surface saturation.
- Refresh banner should feel informational and calm, not urgent-alert red. Use secondary surface plus accent CTA.
- History entrypoints and external links use body color plus underline by default; they are not accent buttons.
- Resolved incidents in history should use subdued neutral-success treatment, not celebratory green fills.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Load latest changes |
| Empty state heading | No active incidents |
| Empty state body | Open and investigating incidents will appear here. Use History to review resolved incidents without disrupting the active queue. |
| Error state | We couldn't load the incident queue. Refresh the queue. If the problem persists, check the host app database connection and Parapet operator query telemetry, then try again. |
| Destructive confirmation | Resolve incident: Resolve this incident? Confirm only after user impact has stopped and the timeline tells the full story. This writes an immutable audit entry. |

### Tone Rules

- Use calm, declarative language.
- Prefer operator-paced wording such as `available`, `review`, `load`, and `history` over feed-style words such as `latest`, `live`, or `auto-updated`.
- Never imply autonomous action by the UI.
- Avoid congratulatory empty states and dashboard filler copy.

---

## State Contract

- Queue loading state uses 6 skeleton rows in the queue pane; do not block the whole workbench with a fullscreen spinner after first mount.
- Empty state appears only when the active queue truly has zero rows, not while loading and not because history holds resolved incidents.
- Refresh banner appears only when background data changed relative to the visible queue snapshot.
- Refresh banner must not cover queue rows, must remain dismissible by taking the explicit refresh action, and must not stack multiple notices.
- Queue pagination controls stay visible at the bottom of the queue pane when more rows exist in either direction.
- When no incident is selected, the detail pane uses the neutral empty copy `Select an incident to review details`.
- Selection state must remain visible with both tinted background and a persistent accent border; hover alone is never enough.
- Attention chips are optional and singular. If no durable attention state exists, render no chip rather than inventing a placeholder.

---

## Accessibility And Motion

- All queue rows, refresh controls, history links, and pagination controls must be keyboard reachable.
- Selected queue row must expose selected-state semantics such as `aria-current="true"` or equivalent.
- Refresh banner button text must remain explicit out of context; do not label it only `Refresh`.
- Focus ring color matches accent and must remain visible against dominant and secondary surfaces.
- Use only two motion patterns: 140ms fade for queue refresh and 160ms slide/fade for mobile route transitions.
- Disable non-essential motion when the user prefers reduced motion.
