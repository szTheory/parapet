# Roadmap: Parapet v0.3

## Phases

- [ ] **Phase 1: Alert Routing & Reception** - Ingest Prometheus alerts and manage Ecto Incident lifecycle automatically.
- [ ] **Phase 2: Runbooks & Automated Mitigations** - Structured runbooks with operator-triggered mitigations directly from the UI.
- [ ] **Phase 3: Notifications & Escalation** - Modular adapters to broadcast incident state changes to external channels.

## Phase Details

### Phase 1: Alert Routing & Reception
**Goal**: System ingests Prometheus alerts and autonomously manages the Ecto Incident lifecycle (creation, correlation, resolution).
**Depends on**: Nothing
**Requirements**: ROUTING-01, ROUTING-02, ROUTING-03, ROUTING-04
**Success Criteria**:
  1. Operator can configure a webhook receiver endpoint that accepts standard Prometheus Alertmanager payloads.
  2. The system automatically creates a durable Ecto Incident when a "firing" alert is received.
  3. Subsequent identical alerts are correlated to the existing open incident to prevent duplicate entries.
  4. The system automatically resolves the incident when a "resolved" webhook is received.
**Plans**: 3 plans
- [ ] 01-01-PLAN.md — Create the webhook receiver endpoint plug and routing logic
- [ ] 01-02-PLAN.md — Implement incident creation and correlation logic for firing alerts
- [ ] 01-03-PLAN.md — Implement incident resolution logic for resolved alerts

### Phase 2: Runbooks & Automated Mitigations
**Goal**: Operators can define structured runbooks with automated mitigation actions and interact with them via the Operator UI.
**Depends on**: Phase 1
**Requirements**: RUNBOOK-01, RUNBOOK-02, RUNBOOK-03, RUNBOOK-04
**Success Criteria**:
  1. Developer can define a runbook using the `Parapet.Runbook` DSL with steps and mitigation actions.
  2. System automatically attaches the correct runbook to an incident based on SLO or alert mappings.
  3. Operator can view the attached runbook interactively on the incident detail page in the UI.
  4. Operator can execute one-click mitigations from the UI, with the execution durably logged via Tool Audit.
**Plans**: 3 plans
- [ ] 02-01-PLAN.md — Create the `Parapet.Runbook` DSL and mapping logic
- [ ] 02-02-PLAN.md — Extend Operator UI to display interactive runbooks
- [ ] 02-03-PLAN.md — Implement and audit one-click mitigations from the UI
**UI hint**: yes

### Phase 3: Notifications & Escalation
**Goal**: System broadcasts incident state changes to external communication channels with rich context and links.
**Depends on**: Phase 1, Phase 2
**Requirements**: NOTIFY-01, NOTIFY-02, NOTIFY-03, NOTIFY-04
**Success Criteria**:
  1. Developer can configure the notification system using modular adapters (`Parapet.Notifier`) for their environment.
  2. System sends a rich Block Kit message to Slack containing incident details and a link to the UI when an incident occurs.
  3. System sends a rich Adaptive Card to Microsoft Teams under similar conditions.
  4. Operator can view a complete audit trail of all dispatched notifications on the incident's Timeline.
**Plans**: 3 plans
- [x] 03-01-PLAN.md — Create `Parapet.Notifier` behavior and timeline integration
- [x] 03-02-PLAN.md — Implement the Slack adapter with Block Kit support
- [x] 03-03-PLAN.md — Implement the MS Teams adapter with Adaptive Cards support
