# Phase 2 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none; use host-owned Phoenix LiveView function components |
| Icon library | Heroicons outline, 20px and 24px only |
| Font | IBM Plex Sans with `ui-sans-serif` fallback |

### Source Notes

- `Tool: none` is locked by project shape: Phoenix LiveView library surface, not React/Next/Vite.
- No first-party UI system, CSS pipeline, or token config exists in the repo as of 2026-05-11.
- UI must stay host-owned and generator-friendly rather than hiding behavior in a packaged admin shell.

---

## Layout Contract

- Primary desktop experience is a three-pane incident workbench.
- Primary focal point on load is the selected incident summary card header: incident title, severity/state pair, impact sentence, and next safe action must read as one visual anchor before the operator scans the timeline or action rail.
- Left pane width: 320px. Contents: queue filters, state counts, incident list.
- Main pane width: fluid, target 720px to 880px readable measure. Contents: incident summary, facts, hypotheses, timeline.
- Right rail width: 320px. Contents: mitigations, approvals, runbook/Grafana links, operator metadata.
- Page max width: 1600px with 32px outer padding on desktop.
- On screens below 960px, collapse to index/detail routes using the same URL-driven incident selection model.
- On mobile detail view, actions move below the summary as stacked cards; do not use a floating FAB.
- Do not introduce primary navigation tabs inside the workbench. Operators must see summary, evidence, and actions in one vertical scan.

---

## Component Inventory

- `IncidentStateCounts`: open, investigating, resolved counts shown above the queue.
- `IncidentQueueRow`: severity, title, affected journey, state, relative update time, approval-needed marker.
- `IncidentSummaryCard`: severity, state, impact sentence, correlated change, next safe action.
- `EvidenceFactsCard`: top facts only, max 5 bullets before “View timeline”.
- `HypothesesCard`: operator-authored hypotheses and latest recommendation.
- `TimelineList`: append-only evidence feed with typed rows.
- `TimelineComposer`: note and hypothesis entry only; no edit affordance on factual system events.
- `ActionRail`: state transition, runbook, change marker, note, approval request, recommendation response.
- `ApprovalPanel`: pending approval request, requester, reason, scope, confirmation action.
- `ExternalLinksCard`: Grafana and runbook links only; no embedded charts or long procedural text.

---

## Interaction Contract

- Selection is URL-driven. Desktop selection updates query params; mobile uses index/detail routes.
- Queue default sort:
  Open and investigating incidents first, ordered by last evidence timestamp descending.
  Resolved incidents second, ordered by resolved timestamp descending.
- The selected incident row uses a persistent accent border and tinted background; hover alone is never the only selection cue.
- Timeline is append-only for factual records. Operators may edit only narrative note title/body and manually entered hypothesis text.
- Every mutating action opens a guided form or confirmation sheet that requires a reason before submit.
- Every mutating action must surface actor, reason, correlation ID, and expected audit outcome before confirmation.
- Approval-gated actions render in two steps: request approval, then approve or reject. Never collapse approval into a single destructive click.
- External links open in a new tab with an external-link icon and short destination label.
- Loading states use skeleton rows in the queue and timeline; never block the whole page with a fullscreen spinner after initial mount.
- Real-time updates may append timeline entries, but they must not steal scroll position if the operator is reading older evidence.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 | 1.5 |
| Label | 14px | 600 | 1.4 |
| Heading | 20px | 600 | 1.25 |
| Display | 28px | 600 | 1.15 |

### Type Rules

- Use `Display` only for the incident title area and queue count headline.
- Use `Heading` for section titles and action-card titles.
- Use `Label` for pills, metadata keys, and action button text.
- Evidence JSON or correlation IDs may use the host app’s default monospace stack at 13px; this is not a fifth scale token for general UI copy.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#F5F2EA` | App background, main reading surface, neutral canvas |
| Secondary (30%) | `#E7E0D3` | Queue pane, right rail, cards, grouped evidence sections |
| Accent (10%) | `#0F766E` | Selected queue row, primary safe CTA, active filter chip, pending-approval badge |
| Destructive | `#B42318` | Destructive confirmations and irreversible mitigation affordances only |

Accent reserved for: selected incident row treatment, `Mark investigating`, `Record note`, active queue filter, pending approval badge. Do not use accent for every link, every badge, or resolved-state decoration.

### Color Rules

- Severity should use icon + label first, with restrained tinting rather than full-surface saturation.
- Resolved state should not use celebratory green fills; use neutral success text treatment only.
- Grafana and runbook links should inherit body color plus underline; they do not get accent-button treatment unless they are the primary action in an empty state.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Mark investigating |
| Empty state heading | No active incidents |
| Empty state body | All current incidents are resolved. Review recent evidence below or open Grafana to inspect background signals before taking action. |
| Error state | We couldn't load incident evidence. Refresh this page. If the problem persists, confirm the host app auth mount and Parapet repo configuration, then try again. |
| Destructive confirmation | Disable feature flag: Disable this flag? This changes live application behavior and writes an immutable audit entry. Type `DISABLE` to continue. Resolve incident: Resolve this incident? Confirm only after user impact has stopped and the timeline tells the full story. |

### Tone Rules

- Use calm, declarative language.
- Prefer evidence wording: facts, hypotheses, recommendation, action taken.
- Never frame the UI as autonomous. Actions are proposed, requested, approved, and recorded.
- Avoid dashboard filler copy such as “everything looks great” or “all systems nominal”.

---

## State Contract

- Queue empty state appears only when there are zero incidents in the current filter, not while loading.
- Summary card always shows severity, state, affected journey, impact sentence, correlated change, and next safe action above the fold.
- Timeline row taxonomy for v1:
  alert fired, state changed, note added, hypothesis added, deploy marker attached, approval requested, approval granted, approval rejected, tool action executed.
- Tool audit details are collapsed by default behind “View audit payload”; never dump raw tool input/output inline in the main evidence scan.
- AI recommendations must show proposer, timestamp, approval status, and explicit accept/reject controls.

---

## Accessibility And Motion

- All queue rows, filters, timeline expanders, and action controls must be keyboard reachable.
- Selected incident row must expose `aria-current="true"` or equivalent selected-state semantics.
- Focus ring color matches accent and must remain visible against both dominant and secondary surfaces.
- Confirmation sheets trap focus until dismissed or completed.
- Use only two motion patterns: 140ms fade for content refresh and 160ms slide/fade for mobile route transitions.
- Disable non-essential motion when the user prefers reduced motion.

---

## Security And Audit UX

- The UI must assume it is already inside a host-authenticated LiveView scope.
- Every action card that mutates state must display “Writes audit record” as a visible secondary label.
- Approval-gated actions must display requester, requested scope, and reason before approval controls appear.
- Never expose raw secret values, access tokens, or full mutation payloads in the operator surface.
- If authorization blocks an action, keep the evidence visible and disable only the action control with explanatory copy.
