# Requirements: Parapet v0.3

## Categories

- [ROUTING] Alert Routing & Reception
- [RUNBOOK] Runbooks & Automated Mitigations
- [NOTIFY] Notifications & Escalation

## Requirements

### Alert Routing & Reception (ROUTING)
- ROUTING-01: System provides a webhook receiver endpoint compatible with Prometheus Alertmanager.
- ROUTING-02: System automatically converts incoming Alertmanager "firing" alerts into durable Ecto Incidents.
- ROUTING-03: System automatically resolves or updates Ecto Incidents when Alertmanager sends a "resolved" webhook.
- ROUTING-04: System correlates incoming alerts to existing open incidents if they share the same alert name and labels, preventing incident spam.

### Runbooks & Automated Mitigations (RUNBOOK)
- RUNBOOK-01: System provides a DSL (`Parapet.Runbook`) to define structured runbooks with explicit steps, descriptions, and required evidence.
- RUNBOOK-02: System allows mapping specific runbooks to specific SLOs or alert names, ensuring they are automatically attached when an incident is created.
- RUNBOOK-03: System Operator UI displays the attached runbook interactively on the Incident detail page.
- RUNBOOK-04: System provides a mechanism for "one-click mitigations" (e.g., executing a server-side callback function) directly from the runbook UI, which must be durably recorded via a Tool Audit log.

### Notifications & Escalation (NOTIFY)
- NOTIFY-01: System provides a modular notification behavior (`Parapet.Notifier`) for broadcasting incident state changes.
- NOTIFY-02: System includes an out-of-the-box Slack adapter that sends rich Block Kit messages containing incident details and a direct link to the Parapet Operator UI.
- NOTIFY-03: System includes an out-of-the-box Microsoft Teams adapter that sends rich Adaptive Cards.
- NOTIFY-04: System durably records all dispatched notifications as Timeline Entries on the incident to maintain a complete audit trail.
