# v0.8 Roadmap: Deterministic Escalation & Bounded Mitigation

## Goal
Implement durable, Oban-backed escalation policies and safe, bounded auto-mitigations using the existing Operator API. This milestone proves that Parapet can take safe action without violating its core "evidence-first, no autonomous AI mutation" brand identity.

## Phases

### Phase 1: Durable Escalation Engine
**Focus:** Build the underlying Oban-backed routing logic for incidents.
- Define `Parapet.Escalation.Policy` behaviour.
- Implement `Parapet.Escalation.Worker` (Oban job).
- Integrate with `Parapet.Evidence.Incident` lifecycle (schedule on creation, short-circuit on acknowledge/resolve).
- Emit timeline entries for escalation events.

### Phase 2: Bounded Runbook Execution
**Focus:** Connect the alert router to the Operator API for system-driven mitigation.
- Extend `Parapet.Runbook` DSL with `auto_execute` configuration.
- Implement the `Parapet.Automation.Executor` that acts as the `:system` identity.
- Route matching incoming alerts directly to the Executor.
- Ensure `ToolAudit` and `TimelineEntry` records are correctly stamped with system identity.

### Phase 3: Circuit Breakers & Flap Protection
**Focus:** Safety guardrails using existing Ecto evidence.
- Implement the `Parapet.Automation.CircuitBreaker` leveraging `ToolAudit` lookbacks.
- Add configuration for `max_executions` and `within` window.
- Plumb circuit-breaker rejections into the escalation engine (if automation fails/blocks, escalate immediately).

### Phase 4: Operator UI Surfacing
**Focus:** Expose the automated actions and pending escalations to humans.
- Add the Escalation Timeline visualizer to the LiveView Incident detail page.
- Style `System-Executed` actions with distinct UI patterns to separate them from human/AI Copilot actions.
- Add manual controls to override or trigger escalations.
