# Plan: 04-03 - Doctor CLI Task

## Goal
Implement `mix parapet.doctor` to perform static analysis health checks and act as a CI gate. (Addresses DOCT-01, DOCT-02, DOCT-03, DOCT-04, DOCT-05).

## Approach
1. **Create Mix Task:** Add `lib/mix/tasks/parapet.doctor.ex`.
2. **Check 1: Security (LiveDashboard & /metrics):** Use `Sourceror` to parse `lib/my_app_web/router.ex`. Look for `live_dashboard` and `forward "/metrics"`. Verify they are wrapped in a `scope` that pipes through an authentication plug (e.g., `:require_authenticated_user`, `RequireAuth`). If not found, emit a `[warn]`.
3. **Check 2: Label Cardinality:** We need a way to check if the user has defined custom metrics with bad labels. This might require analyzing `lib/my_app_web/telemetry.ex` or `Parapet.Instrumenter` for `tags: [:user_id, ...]` or similar high-cardinality flags. Alternatively, we rely on the compile-time checks from Phase 1 (TELE-03) and use the Doctor to just verify the `Parapet.Instrumenter` is still properly attached to the Endpoint.
4. **Check 3: Runbooks:** Inspect all configured SLOs (either by looking at the `Parapet.SLO` definitions or the generated prometheus rules) to ensure a `runbook:` URL/path is present. (SLO-05).
5. **Output Formatting:** Implement standard colorful output for human terminals, and structured JSON output when `--ci` is passed.
6. **Exit Codes:** Return `0` for all pass, `1` for warnings (like missing auth plug), `2` for severe violations (like missing runbooks on critical SLOs).

## Success Criteria
- Task runs in < 5 seconds without starting the full app tree (using static analysis).
- Correctly identifies unsecured `/metrics` or `live_dashboard` routes via AST parsing.
- Exits with appropriate codes (0, 1, 2).
- Emits JSON with `--ci`.
