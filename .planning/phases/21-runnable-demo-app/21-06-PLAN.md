Repo: `szTheory/parapet`, default branch `main`.

PRECONDITION: the `release_gate` check name only becomes selectable in GitHub branch
protection AFTER the workflow has run at least once on a PR or commit on `main`. If
`release_gate` has never run, the API call below will succeed structurally but the check name
will not be registered as an available check until a workflow run reports it. Run the workflow
once first (push a commit or open a PR that triggers CI) before configuring, or accept that
the protection rule will reference a check name that activates on the next run.

CONFIGURATION COMMAND (full protection payload — the robust form):
  gh api -X PUT repos/szTheory/parapet/branches/main/protection --input - <<'EOF'
  {
    "required_status_checks": {
      "strict": false,
      "checks": [{"context": "release_gate", "app_id": null}]
    },
    "enforce_admins": false,
    "required_pull_request_reviews": null,
    "restrictions": null
  }
  EOF

CONFIRMATION COMMAND:
  gh api repos/szTheory/parapet/branches/main/protection/required_status_checks --jq '.checks'
Expected output includes an entry whose `context` is `release_gate`.

NOTE: the narrower `gh api -X PUT .../required_status_checks -f 'checks[][context]=release_gate'`
form exists but may reset other protection settings; prefer the full-payload PUT above. This
is a human-required action: it modifies GitHub repository settings outside the codebase, may
require admin auth on `gh`, and the exact moment the check name is registered depends on a
prior workflow run.
