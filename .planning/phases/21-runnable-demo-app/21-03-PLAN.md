<!-- Verified against the parapet codebase. -->

Parapet.Evidence Stable API (lib/parapet/evidence.ex):
  create_incident(attrs \\ %{}) :: {:ok, %Incident{}} | {:error, changeset}
  append_timeline(incident_id, attrs \\ %{}) :: {:ok, %TimelineEntry{}} | {:error, ...}
  log_tool_audit(attrs \\ %{}) :: {:ok, %ToolAudit{} | :deferred} | {:error, ...}

Incident CASTABLE FIELDS ONLY (lib/parapet/spine/incident.ex:55):
  :title (required), :description, :state, :correlation_key, :trace_id, :runbook_data
  state must be one of "open" | "investigating" | "resolved" (validate_inclusion).
  THERE IS NO `source` OR `alert_name` FIELD. CONTEXT.md D-06 mentions them but they do not
  exist — put that info in title/description instead. Ecto silently drops unknown attrs.

runbook_data MUST use STRING KEYS (Ecto :map → JSONB round-trips atoms to strings).
At least one step must have a non-nil "warning" (DEMO-02). Use the shape from RESEARCH.md
Pattern 5 (lines 681-718) / PATTERNS.md seeds section (lines 366-426): a map with
"title", "description", and "steps" (list of step maps, each with string keys including
"warning").

TimelineEntry.changeset requires :type (string) and accepts :payload (map). Use types like
"note" and "status_change" with a "text"/"new_state" payload (RESEARCH.md Pattern 5).
ToolAudit.changeset requires :tool_name, :input, :success; accepts :output, :duration_ms.

Repo registration: config/config.exs already sets `config :parapet, repo: DemoApp.Repo`
(Plan 01). seeds.exs adds belt-and-suspenders `Application.put_env(:parapet, :repo,
DemoApp.Repo)` so it is safe to run standalone (PATTERNS.md lines 370-382).

Smoke test pattern (RESEARCH.md Pattern 3, lines 606-648):
  - test/support/conn_case.ex = DemoAppWeb.ConnCase using ExUnit.CaseTemplate, `using do
    use Phoenix.ConnTest; @endpoint DemoAppWeb.Endpoint end`, setup checks out the Ecto
    sandbox and returns {:ok, conn: Phoenix.ConnTest.build_conn()}.
  - test/test_helper.exs = `ExUnit.start()` + `Ecto.Adapters.SQL.Sandbox.mode(DemoApp.Repo,
    :manual)`.
  - The smoke test uses `@moduletag :smoke` so `mix test --only smoke` selects the whole file.

CRITICAL — sandbox vs seeds (RESEARCH.md Pitfall 3, lines 813-822): In the SANDBOX test env,
the seeds run via `mix run priv/repo/seeds.exs` are NOT visible inside a sandboxed test
transaction. The smoke test's "incident count > 0" must therefore create its own incident
inside the test (via Parapet.Evidence.create_incident in a setup/test that shares the sandbox
connection) OR the conn_case must run in shared sandbox mode and the test inserts a row
before asserting. Choose the robust approach: have the smoke test (or its setup) call
`Parapet.Evidence.create_incident/1` to insert at least one incident within the sandboxed
connection, then assert count > 0. This makes the smoke test self-contained and independent
of whether `mix run priv/repo/seeds.exs` populated the (separate, non-sandbox) dev DB. The CI
job (Plan 04) still runs seeds for the dev DB so `mix phx.server` shows data, but the smoke
test must not depend on cross-connection seed visibility.
