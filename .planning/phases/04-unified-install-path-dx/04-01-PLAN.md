<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| maintainer CLI input -> Igniter task | flags and prompt answers decide what code/config gets written into the host app |
| installer task -> host source tree | generated code and config become durable host-owned runtime behavior |
| dependency detection -> optional integration enablement | detected libraries are advisory signals only and must not silently widen the runtime surface |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | T | `lib/mix/tasks/parapet.install.ex` | mitigate | Keep every optional branch behind explicit flags or a single high-impact prompt; do not infer permission from dependency detection, per D-12 through D-14. |
| T-04-02 | E | generated UI install path | mitigate | Gate `parapet.gen.ui` on LiveView presence and reuse the existing auth notice so the installer never implies Parapet owns router authentication, per D-04. |
| T-04-03 | T | optional integration wiring | mitigate | Generate only `Parapet.attach(adapters: [...])` and `config :parapet, providers: [...]` host-owned changes; never mutate `mix.exs` or invent a second activation path, per D-15 and D-16. |
| T-04-04 | R | installer summary surface | mitigate | Emit an end-of-run notice listing generated files, selected/skipped extras, host follow-up, and `mix parapet.doctor` so maintainers can audit what the task did, per D-06 and D-11. |
</threat_model>
